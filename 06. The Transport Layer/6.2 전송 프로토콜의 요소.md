# 6.2 전송 프로토콜의 요소 (Elements of Transport Protocols)

전송 서비스(transport service)는 두 전송 엔티티(transport entity) 사이에서 사용되는 **전송 프로토콜(transport protocol)** 에 의해 구현됩니다. 여러 면에서 전송 프로토콜은 3장에서 자세히 공부한 데이터 링크 프로토콜(data link protocol)과 유사합니다. 두 프로토콜 모두 오류 제어(error control), 순서화(sequencing), 흐름 제어(flow control) 등의 문제를 다루어야 합니다.

그러나 두 프로토콜 사이에는 중요한 차이점이 존재합니다. 이러한 차이점은 두 프로토콜이 동작하는 환경의 근본적인 차이에서 비롯됩니다. 이 차이는 그림 6-7(Figure 6-7)에 나타나 있습니다.

<img width="675" height="235" alt="Image" src="https://github.com/user-attachments/assets/a599e10b-bb08-4662-b509-7123871cf7d4" />

데이터 링크 계층에서는 두 라우터가 유선이든 무선이든 물리적 채널(physical channel)을 통해 직접 통신하는 반면, 전송 계층에서는 이 물리적 채널이 전체 네트워크로 대체됩니다. 이 차이는 프로토콜에 많은 중요한 영향을 미칩니다.

첫째, 유선이나 광섬유와 같은 점대점 링크(point-to-point link)에서는 라우터가 어떤 라우터와 통신할지 명시할 필요가 없습니다. 각 나가는 회선이 특정 라우터로 직접 연결되기 때문입니다. 그러나 전송 계층에서는 목적지의 명시적인 주소 지정(explicit addressing)이 필요합니다.

둘째, 그림 6-7(a)에서처럼 유선을 통한 연결을 설정하는 과정은 단순합니다. 상대방이 항상 존재하기 때문입니다(물론 고장 난 경우는 제외). 무선 링크에서도 과정은 크게 다르지 않습니다. 메시지를 보내기만 하면 모든 목적지에 도달할 수 있고, 오류로 인해 확인 응답(acknowledgement)이 되지 않으면 재전송할 수 있습니다. 그러나 전송 계층에서는 초기 연결 설정(initial connection establishment)이 복잡합니다.

셋째, 데이터 링크 계층과 전송 계층 사이의 또 다른 (매우 성가신) 차이점은 네트워크 내에 저장 용량(storage capacity)이 존재할 가능성입니다. 라우터가 링크를 통해 패킷을 보내면 패킷이 도착하거나 손실될 수는 있지만, 한동안 돌아다니다가 세계 어딘가에 숨어 있다가 다른 패킷보다 훨씬 늦게 갑자기 나타날 수는 없습니다. 네트워크가 독립적으로 라우팅되는 데이터그램(datagram)을 사용하는 경우, 패킷이 경치 좋은 경로(scenic route)를 타고 늦게 도착하거나, 예상 순서와 다르게 도착하거나, 심지어 패킷의 복사본(duplicate)이 도착할 수도 있는 무시할 수 없는 확률이 존재합니다. 네트워크가 패킷을 지연시키고 복제하는 능력의 결과는 때때로 치명적일 수 있으며, 전송 정보를 올바르게 처리하기 위해 특별한 프로토콜이 필요할 수 있습니다.

넷째, 데이터 링크 계층과 전송 계층 사이의 마지막 차이점은 종류가 아닌 정도(degree)의 차이입니다. 버퍼링(buffering)과 흐름 제어(flow control)는 두 계층 모두에서 필요하지만, 전송 계층에서는 대규모이고 가변적인 수의 연결이 존재하며, 각 연결이 서로 대역폭(bandwidth)을 경쟁하므로 데이터 링크 계층에서 사용한 것과는 다른 접근 방식이 필요할 수 있습니다. 3장에서 논의한 일부 프로토콜은 각 회선에 고정된 수의 버퍼를 할당하여 프레임이 도착하면 항상 버퍼가 사용 가능하도록 합니다. 전송 계층에서는 관리해야 할 연결 수가 더 많고 각 연결이 받는 대역폭의 변동이 크기 때문에, 많은 버퍼를 각 연결에 할당하는 것은 매력적이지 않습니다.

다음 섹션에서는 이러한 모든 중요한 문제들을 살펴보겠습니다.

## 6.2.1 주소 지정 (Addressing)

애플리케이션(application), 즉 사용자 프로세스(user process)가 원격 애플리케이션 프로세스에 대한 연결을 설정하고자 할 때, 어떤 프로세스에 연결할 것인지를 명시해야 합니다. (비연결형 전송(connectionless transport)도 동일한 문제를 가지고 있습니다: 각 메시지가 누구에게 전송되어야 하는지의 문제입니다.) 일반적으로 사용되는 방법은 프로세스가 연결 요청(connection request)을 수신할 수 있는 전송 주소(transport address)를 정의하는 것입니다. 인터넷에서 이러한 종단점(endpoint)을 **포트(port)** 라고 부릅니다. 여기서는 일반적인 용어인 **TSAP** (**Transport Service Access Point**, 전송 서비스 접근점)을 사용하여 전송 계층의 특정 종단점을 의미하겠습니다. 네트워크 계층의 유사한 종단점(즉, 네트워크 계층 주소)은 당연히 **NSAP** (**Network Service Access Point**, 네트워크 서비스 접근점)이라고 불립니다. IP 주소(IP address)는 NSAP의 예입니다.

애플리케이션 프로세스(클라이언트와 서버 모두)는 자신을 로컬 TSAP에 연결하여 원격 TSAP에 대한 연결을 설정할 수 있습니다. 이 연결들은 각 호스트의 NSAP를 통해 이루어집니다. TSAP를 갖는 목적은 일부 네트워크에서 각 컴퓨터가 단일 NSAP를 가지므로, 해당 NSAP를 공유하는 여러 전송 종단점을 구별할 방법이 필요하기 때문입니다.

### 전송 연결의 가능한 시나리오

전송 연결의 가능한 시나리오는 다음과 같습니다:

1. Host 2의 메일 서버(mail server) 프로세스가 TSAP 1522에 자신을 연결하여 들어오는 호출(incoming call)을 기다립니다. 프로세스가 TSAP에 자신을 연결하는 방법은 네트워킹 모델 외부에 있으며 전적으로 로컬 운영 체제(operating system)에 따라 다릅니다. 예를 들어 LISTEN과 같은 호출이 사용될 수 있습니다.

2. Host 1의 애플리케이션 프로세스가 이메일 메시지를 보내고자 하여, TSAP 1208에 자신을 연결하고 CONNECT 요청을 발행합니다. 이 요청은 Host 1의 TSAP 1208을 출발지(source)로, Host 2의 TSAP 1522를 목적지(destination)로 지정합니다. 이 동작은 궁극적으로 애플리케이션 프로세스와 서버 사이에 전송 연결이 설정되는 결과를 가져옵니다.

3. 애플리케이션 프로세스가 메일 메시지를 전송합니다.

4. 메일 서버가 메시지를 전달하겠다고 응답합니다.

5. 전송 연결이 해제(release)됩니다.

Host 2에는 동일한 NSAP를 통해 도착하는 들어오는 연결을 위해 다른 TSAP에 연결되어 기다리고 있는 다른 서버들이 있을 수 있다는 점에 유의하세요.

### TSAP 주소를 알아내는 방법

위의 시나리오는 좋지만, 하나의 작은 문제가 은밀히 숨어 있습니다: Host 1의 사용자 프로세스는 메일 서버가 TSAP 1522에 연결되어 있다는 것을 어떻게 아는 것일까요?

#### 방법 1: 안정적인 TSAP 주소 (Stable TSAP Addresses)

한 가지 가능성은 메일 서버가 수년 동안 TSAP 1522에 연결되어 있어서 점차 모든 네트워크 사용자가 이를 알게 되는 것입니다. 이 모델에서는 서비스가 잘 알려진 장소(well-known places)에 나열된 안정적인 TSAP 주소를 가지고 있습니다. 예를 들어, UNIX 시스템의 */etc/services* 파일은 어떤 서버가 어떤 포트에 영구적으로 연결되어 있는지를 나열하며, 메일 서버가 TCP 포트 25에서 발견된다는 사실도 포함합니다.

#### 방법 2: 포트매퍼 (Portmapper)

안정적인 TSAP 주소는 변하지 않는 소수의 핵심 서비스(예: 웹 서버)에는 효과적이지만, 일반적으로 사용자 프로세스는 미리 알려지지 않았거나 짧은 시간 동안만 존재할 수 있는 TSAP 주소를 가진 다른 사용자 프로세스와 통신하고 싶어합니다.

이 상황을 처리하기 위해 대안적인 방식이 사용될 수 있습니다. 이 방식에서는 **포트매퍼(portmapper)** 라는 특별한 프로세스가 존재합니다. 주어진 서비스 이름(예: "BitTorrent")에 해당하는 TSAP 주소를 찾기 위해, 사용자는 포트매퍼(잘 알려진 TSAP에서 수신 대기)에 연결을 설정합니다. 그런 다음 사용자는 서비스 이름을 명시하는 메시지를 보내고, 포트매퍼는 TSAP 주소를 돌려줍니다. 사용자는 포트매퍼와의 연결을 해제하고 원하는 서비스와 새로운 연결을 설정합니다.

이 모델에서 새로운 서비스가 생성되면, 서비스 이름(일반적으로 ASCII 문자열)과 TSAP를 모두 제공하여 포트매퍼에 자신을 등록해야 합니다. 포트매퍼는 이 정보를 내부 데이터베이스(internal database)에 기록하여 이후 질의에 응답할 수 있습니다.

포트매퍼의 기능은 전화 시스템의 안내 전화 교환원(directory assistance operator)과 유사합니다 -- 이름을 번호로 매핑해 줍니다. 전화 시스템에서와 마찬가지로, 포트매퍼가 사용하는 잘 알려진 TSAP의 주소가 실제로 잘 알려져 있어야 합니다. 안내 전화 교환원의 번호를 모르면 전화를 걸어 찾을 수 없습니다.

#### 방법 3: 초기 연결 프로토콜 (Initial Connection Protocol)

머신에 존재할 수 있는 많은 서버 프로세스는 드물게만 사용됩니다. 각각을 항상 활성화하고 안정적인 TSAP 주소에서 수신 대기하게 하는 것은 낭비입니다. 대안적인 방식이 그림 6-9(Figure 6-9)에 간략한 형태로 나타나 있습니다. 이것은 **초기 연결 프로토콜(initial connection protocol)** 로 알려져 있습니다.

<img width="698" height="522" alt="Image" src="https://github.com/user-attachments/assets/00ddee9b-5333-4f04-9add-4646da379be0" />

모든 가능한 서비스마다 서버를 수신 대기시키는 대신, 원격 사용자에게 서비스를 제공하려는 각 머신에는 **프로세스 서버(process server)** 라는 특별한 서버가 있습니다. 이 서버는 덜 많이 사용되는 서버의 대리(proxy) 역할을 합니다. UNIX 시스템에서 이 서버는 *inetd*라고 불립니다. 이 서버는 여러 포트 집합을 동시에 수신 대기하며 연결 요청을 기다립니다. 서비스의 잠재적 사용자는 원하는 서비스의 TSAP 주소를 지정하여 CONNECT 요청을 하여 시작합니다. 해당 서비스를 위해 기다리고 있는 서버가 없으면, 그림 6-9(a)에 표시된 것처럼 프로세스 서버에 연결됩니다.

들어오는 요청을 받으면 프로세스 서버는 요청된 서버를 생성(spawn)하고, 사용자와의 기존 연결을 상속(inherit)하게 합니다. 새 서버가 요청된 작업을 수행하는 동안, 프로세스 서버는 그림 6-9(b)에 표시된 것처럼 새로운 요청을 위해 수신 대기로 돌아갑니다. 이 방법은 서버가 요청 시에만 생성될 수 있는 경우에만 적용 가능합니다.

## 6.2.2 연결 설정 (Connection Establishment)

연결을 설정하는 것은 간단해 보이지만, 실제로는 놀라울 정도로 까다롭습니다(tricky). 언뜻 보면 한 전송 엔티티가 목적지에 CONNECTION REQUEST 세그먼트를 보내고 CONNECTION ACCEPTED 응답을 기다리면 충분할 것 같습니다. 문제는 네트워크가 패킷을 손실(lose), 지연(delay), 손상(corrupt), 그리고 복제(duplicate)할 수 있을 때 발생합니다. 이러한 동작은 심각한 복잡성을 야기합니다.

### 지연된 중복 패킷의 위험

네트워크가 매우 혼잡(congested)하여 확인 응답(acknowledgement)이 거의 돌아오지 않고 각 패킷이 두세 번 재전송(retransmit)되는 상황을 상상해 보세요. 네트워크가 데이터그램(datagram)을 사용하고 모든 패킷이 다른 경로를 따른다고 가정합니다. 일부 패킷이 네트워크 내부의 교통 체증에 빠져 도착하는 데 오랜 시간이 걸릴 수 있습니다. 즉, 패킷이 네트워크 내에서 지연되었다가, 송신자가 패킷이 손실되었다고 판단한 한참 후에 나타날 수 있습니다.

최악의 악몽(worst possible nightmare)은 다음과 같습니다: 사용자가 은행과 연결을 설정하고, 완전히 신뢰할 수 없는 사람의 계좌로 큰 금액을 이체하라는 메시지를 보냅니다. 불행히도 패킷들은 경치 좋은 경로를 타고 네트워크의 먼 구석을 탐험하러 갑니다. 송신자가 타임아웃(timeout)을 하고 패킷들을 모두 다시 보냅니다. 이번에는 패킷들이 최단 경로를 타고 빠르게 전달되어 송신자가 연결을 해제합니다.

불행히도 결국 처음 보낸 패킷들이 숨어 있다가 나와서 목적지에 순서대로 도착합니다. 은행에 새 연결을 설정하고 (다시) 돈을 이체하라고 요청합니다. 은행은 이것이 중복(duplicate)이라고 판별할 방법이 없습니다. 이것을 두 번째 독립적인 거래라고 가정하고 돈을 다시 이체합니다.

이 시나리오는 있을 법하지 않거나 심지어 터무니없어 보일 수 있지만, 요점은 이것입니다: **프로토콜은 모든 경우에 올바르도록 설계되어야 합니다.** 일반적인 경우에만 효율적으로 구현하면 되지만, 프로토콜은 드문 경우에도 문제없이 대처할 수 있어야 합니다. 그렇지 않으면 조건이 나빠질 때 경고 없이 실패하는 순풍에만 작동하는 네트워크(fair-weather network)를 구축한 것입니다.

이 섹션의 나머지 부분에서는 지연된 중복(delayed duplicate) 문제를 연구하며, 위와 같은 악몽이 발생하지 않도록 연결을 안정적으로 설정하는 알고리즘에 중점을 둡니다. 문제의 핵심은 지연된 중복이 새로운 패킷으로 오인되는 것입니다. 패킷이 복제되고 지연되는 것을 방지할 수는 없지만, 만약 그런 일이 발생하면 해당 패킷을 중복으로 거부하고 새로운 패킷으로 처리하지 않아야 합니다.

### 지연된 중복 문제의 해결 접근법

이 문제는 여러 방법으로 공격할 수 있지만, 어느 것도 완전히 만족스럽지는 않습니다.

#### 방법 1: 일회용 전송 주소 (Throwaway Transport Addresses)

한 가지 방법은 일회용 전송 주소를 사용하는 것입니다. 이 접근법에서는 전송 주소가 필요할 때마다 새로운 것이 생성됩니다. 연결이 해제되면 주소는 폐기되고 다시 사용되지 않습니다. 지연된 중복 패킷은 전송 프로세스에 도달하지 못하므로 피해를 줄 수 없습니다. 그러나 이 접근법은 프로세스에 처음 연결하는 것을 더 어렵게 만듭니다.

#### 방법 2: 연결 식별자 (Connection Identifier)

또 다른 가능성은 각 연결에 고유한 식별자(unique identifier), 즉 연결이 설정될 때마다 개시자(initiating party)가 선택하여 증가시키는 순서 번호(sequence number)를 부여하고, 각 세그먼트(연결을 요청하는 것 포함)에 넣는 것입니다. 각 연결이 해제된 후, 각 전송 엔티티는 (상대방 전송 엔티티, 연결 식별자) 쌍을 나열하는 폐기된 연결 테이블(table)을 업데이트할 수 있습니다. 연결 요청이 들어오면 테이블을 확인하여 이전에 해제된 연결에 속하는지 여부를 확인할 수 있습니다.

불행히도 이 방식에는 근본적인 결함이 있습니다: 각 전송 엔티티가 일정량의 이력 정보(history information)를 무한히 유지해야 합니다. 이 이력은 출발지와 목적지 양쪽 머신에 지속되어야 합니다. 그렇지 않으면 머신이 고장(crash)나서 메모리를 잃으면 상대방이 이미 사용한 연결 식별자가 무엇인지 더 이상 알 수 없게 됩니다.

### 패킷 수명 제한 (Restricting Packet Lifetime)

대신, 문제를 단순화하기 위해 다른 방법을 취합니다. 패킷이 네트워크 내에서 영원히 살아있도록 허용하는 대신, 여전히 떠돌아다니는 오래된 패킷을 제거하는 메커니즘을 고안합니다. 이 제한이 적용되면 문제는 다소 관리 가능해집니다.

패킷 수명(packet lifetime)은 다음 기법 중 하나 이상을 사용하여 알려진 최대값으로 제한할 수 있습니다:

1. **제한된 네트워크 설계 (Restricted network design)** : 패킷이 루핑(looping)하는 것을 방지하는 모든 방법을 포함하며, (현재 알려진) 가장 긴 경로를 포함한 혼잡(congestion)을 통한 지연을 제한하는 방법과 결합됩니다. 인터넷의 범위가 단일 도시에서 국제적 규모까지 다양할 수 있다는 점을 고려하면 어렵습니다.

2. **홉 카운터 (Hop counter)** : 각 패킷에 홉 카운터를 넣는 것입니다. 적절한 값으로 초기화하고 패킷이 포워딩될 때마다 감소시킵니다. 네트워크 프로토콜은 홉 카운터가 0이 된 패킷을 단순히 폐기합니다.

3. **타임스탬핑 (Timestamping)** : 각 패킷에 타임스탬프를 넣는 것입니다. 라우터들이 합의된 시간보다 오래된 패킷을 폐기하도록 동의합니다. 이 방법은 라우터 시계가 동기화(synchronized)되어야 하는데, 이는 쉽지 않은 작업이며, 실제로 홉 카운터는 패킷 수명(age)에 대한 근사치로 사용됩니다.

실제로는 패킷이 죽었을 뿐만 아니라 그에 대한 모든 확인 응답(acknowledgement)도 죽었다는 것을 보장해야 합니다. 따라서 이제 기간 **T**를 도입합니다. T는 실제 최대 패킷 수명의 작은 배수(small multiple)입니다. 최대 패킷 수명은 네트워크에 대한 보수적인 상수(conservative constant)입니다. 인터넷에서는 다소 임의적으로 **120초**로 정해져 있습니다. 배수는 프로토콜에 따라 다르며, 단순히 T를 더 크게 만드는 효과가 있습니다. 패킷이 전송된 후 시간 T를 기다리면 패킷의 모든 흔적이 사라졌으며, 패킷 자체도 그 확인 응답도 갑자기 나타나 문제를 일으키지 않을 것이라고 확신할 수 있습니다.

### Tomlinson의 시계 기반 방법 (Clock-Based Method)

패킷 수명이 제한되면, 지연된 중복 세그먼트를 거부하는 실용적이고 완벽한 방법을 고안할 수 있습니다. 아래 설명하는 방법은 Tomlinson(1975)에 의해 제안되었으며, Sunshine과 Dalal(1978)에 의해 개선되었습니다. 이 변형들은 TCP를 포함하여 실제로 널리 사용됩니다.

이 방법의 핵심은 출발지(source)가 세그먼트에 T초 이내에 재사용되지 않는 순서 번호(sequence number)를 부여하는 것입니다. 기간 T와 초당 패킷 속도(rate of packets per second)가 순서 번호의 크기를 결정합니다. 이 방식으로 주어진 순서 번호를 가진 패킷은 어느 시점에서든 단 하나만 존재할 수 있습니다(outstanding). 이 패킷의 복사본은 여전히 발생할 수 있으며 목적지에서 폐기되어야 하지만, 오래된 패킷의 지연된 중복이 같은 순서 번호를 가진 새 패킷을 이기고 목적지에서 대신 수락되는 경우는 더 이상 발생하지 않습니다.

#### 머신 재시작 후의 문제 해결

머신이 모든 메모리를 잃고 어디에 있었는지를 기억하지 못하는 문제를 해결하기 위해, 한 가지 가능성은 충돌(crash) 후 복구(recovery) 후에 전송 엔티티를 T초 동안 유휴(idle) 상태로 두어 모든 오래된 세그먼트가 소멸되도록 하는 것입니다. 그러면 송신자는 어떤 순서 번호로든 다시 시작할 수 있습니다. 그러나 복잡한 인터네트워크에서 T가 클 수 있으므로 이 전략은 매력적이지 않습니다.

대신 Tomlinson은 각 호스트에 **시간 기반 시계(time-of-day clock)** 를 장착할 것을 제안했습니다. 다른 호스트의 시계와 동기화될 필요가 없습니다. 각 시계는 일정한 간격으로 증가하는 이진 카운터(binary counter)의 형태를 취하는 것으로 가정합니다. 또한 카운터의 비트 수는 순서 번호의 비트 수 이상이어야 합니다. 마지막으로, 가장 중요한 것은 **호스트가 다운되더라도 시계가 계속 작동(continue running)** 하는 것으로 가정합니다.

연결이 설정되면 시계의 하위(low-order) **k** 비트가 **k**비트 초기 순서 번호(initial sequence number)로 사용됩니다. 따라서 3장의 프로토콜과 달리, 각 연결은 서로 다른 초기 순서 번호로 번호 매기기를 시작합니다. 순서 번호 공간은 순서 번호가 순환(wrap around)할 때까지 같은 순서 번호를 가진 오래된 세그먼트가 이미 사라질 만큼 충분히 커야 합니다.

금지 영역(forbidden region)은 세그먼트 순서 번호가 사용이 허용되기 전의 시간을 보여줍니다. 이 영역에서 순서 번호를 가진 세그먼트가 전송되면, 지연되어 같은 순서 번호로 약간 후에 발행될 다른 패킷으로 위장할 수 있습니다. 예를 들어, 호스트가 시간 70초에 충돌하고 재시작하면, 시계를 기반으로 초기 순서 번호를 사용하므로 금지 영역에서 더 낮은 순서 번호로 시작하지 않습니다.

실제로 초기 순서 번호 곡선(굵은 선으로 표시)은 선형이 아니라 계단(staircase) 형태입니다. 시계가 이산적인 단계(discrete steps)로 진행하기 때문입니다. 단순화를 위해 이 세부 사항은 무시합니다.

### 금지 영역 외부 유지를 위한 두 가지 주의점

패킷 순서 번호를 금지 영역 밖에 유지하려면 두 가지 면에서 주의해야 합니다. 두 가지 다른 방식으로 문제에 빠질 수 있습니다:

1. **위에서 금지 영역에 진입하는 경우**: 호스트가 새로 열린 연결에서 너무 많은 데이터를 너무 빠르게 보내면, 실제 순서 번호 대 시간 곡선이 초기 순서 번호 대 시간 곡선보다 더 가파르게 상승하여 금지 영역에 진입할 수 있습니다. 이를 방지하기 위해, 모든 연결에서의 최대 데이터 전송률은 **클록 틱당 하나의 세그먼트(one segment per clock tick)** 입니다. 이는 또한 전송 엔티티가 충돌 재시작(crash restart) 후 새 연결을 열기 전에 클록 틱을 기다려야 함을 의미합니다. 같은 번호가 두 번 사용되지 않도록 하기 위해서입니다. 이 두 점 모두 짧은 클록 틱(1마이크로초 이하)을 사용하는 것이 유리합니다. 그러나 시계가 순서 번호에 비해 너무 빨리 틱하면 안 됩니다. 클록 속도 **C**와 크기 **S**의 순서 번호 공간에 대해, 순서 번호가 너무 빨리 순환하지 않도록 **S/C > T**여야 합니다.

2. **아래에서(왼쪽에서) 금지 영역에 진입하는 경우**: 그림 6-10(b)에서 볼 수 있듯이, 클록 속도보다 낮은 데이터 전송률에서는, 시간에 대한 실제 순서 번호의 곡선이 순서 번호가 순환(wrap around)하면서 왼쪽에서 금지 영역에 진입합니다. 실제 순서 번호의 기울기가 클수록 이 이벤트는 더 오래 지연됩니다. 이 상황을 피하는 것은 순서 번호가 연결에서 얼마나 느리게 진행할 수 있는지(또는 연결이 얼마나 오래 지속될 수 있는지)를 제한합니다.

### 세 방향 핸드셰이크 (Three-Way Handshake)

시계 기반 방법은 지연된 중복 세그먼트와 새 세그먼트를 구별하는 문제를 해결합니다. 그러나 연결 설정에 이를 사용하는 데는 실용적인 문제가 있습니다. 우리는 일반적으로 목적지에서 연결 간에 순서 번호를 기억하지 않으므로, CONNECTION REQUEST 세그먼트에 포함된 초기 순서 번호가 최근 연결의 중복인지 여부를 알 수 없습니다. 이 문제는 슬라이딩 윈도우 프로토콜이 현재 순서 번호를 기억하기 때문에 연결 중에는 존재하지 않습니다.

이 특정 문제를 해결하기 위해 Tomlinson(1975)은 **세 방향 핸드셰이크(three-way handshake)** 를 도입했습니다. 이 설정 프로토콜은 상대방에게 연결 요청이 현재 것인지 확인하는 과정을 포함합니다.

Host 1이 개시할 때의 정상 설정 절차는 그림 6-11(a)(Figure 6-11(a))에 나타나 있습니다:

1. Host 1이 순서 번호 **x**를 선택하고, Host 2에 순서 번호 x를 포함하는 CONNECTION REQUEST 세그먼트를 보냅니다.
2. Host 2는 **x**를 확인하고 자신의 초기 순서 번호 **y**를 알리는 ACK 세그먼트(seq = y, ACK = x)로 응답합니다.
3. 마지막으로 Host 1이 첫 번째 데이터 세그먼트에서 Host 2의 초기 순서 번호 선택을 확인합니다(DATA, seq = x, ACK = y).

#### 지연된 중복 CONNECTION REQUEST: 그림 6-11(b)

그림 6-11(b)(Figure 6-11(b))에서 첫 번째 세그먼트는 이전 연결의 지연된 중복 CONNECTION REQUEST입니다. 이 세그먼트는 Host 1의 지식 없이 Host 2에 도착합니다. Host 2는 이 세그먼트에 Host 1에게 ACK 세그먼트를 보내 확인을 요청하여 반응합니다. Host 1이 Host 2의 새 연결 설정 시도를 거부(reject)하면, Host 2는 지연된 중복에 속은 것을 깨닫고 연결을 포기합니다. 이러한 방식으로 지연된 중복은 피해를 주지 않습니다.

#### 지연된 중복 CONNECTION REQUEST와 ACK: 그림 6-11(c)

최악의 경우는 지연된 CONNECTION REQUEST와 ACK가 모두 서브넷에서 떠돌고 있는 경우입니다. 이 경우는 그림 6-11(c)(Figure 6-11(c))에 나타나 있습니다.

> **그림 6-11 설명**: 세 방향 핸드셰이크를 사용한 연결 설정의 세 가지 프로토콜 시나리오. CR은 CONNECTION REQUEST를 나타냅니다. (a) 정상 동작. Host 1이 CR(seq = x)을 보내고, Host 2가 ACK(seq = y, ACK = x)로 응답하며, Host 1이 DATA(seq = x, ACK = y)를 보냅니다. (b) 오래된 중복 CONNECTION REQUEST가 나타남. Host 2가 오래된 CR(seq = x)을 받고 ACK(seq = y, ACK = x)를 보내지만, Host 1이 REJECT(ACK = y)로 거부합니다. (c) 중복 CONNECTION REQUEST와 중복 ACK. Host 2가 오래된 CR(seq = x)을 받고 ACK(seq = y, ACK = x)를 보내며, 오래된 중복 ACK(seq = x, ACK = z)도 도착합니다. Host 2는 DATA(seq = x, ACK = z)에서 z가 y와 다르므로 이것이 오래된 중복임을 인식하고 REJECT(ACK = y)합니다.

이전 예와 마찬가지로 Host 2는 지연된 CONNECTION REQUEST를 받고 응답합니다. 이 시점에서 중요한 것은 Host 2가 **y**를 초기 순서 번호로 제안했다는 것을 인식하는 것입니다. 순서 번호 y를 포함하는 세그먼트나 y에 대한 확인 응답이 아직 존재하지 않는다는 것을 완전히 알고 있습니다. 두 번째 지연된 세그먼트가 Host 2에 도착하면, **z**가 확인(acknowledged)된 것이 **y**가 아닌 것이 Host 2에게 이것도 오래된 중복이라고 알려줍니다. 여기서 인식해야 할 중요한 점은 **아무도 원치 않을 때 우연히 연결이 설정되도록 하는 오래된 세그먼트의 조합은 존재하지 않는다**는 것입니다.

### TCP에서의 세 방향 핸드셰이크

TCP는 이 세 방향 핸드셰이크를 사용하여 연결을 설정합니다. 연결 내에서 타임스탬프(timestamp)가 32비트 순서 번호를 확장하는 데 사용되어, 기가비트 초당(gigabit-per-second) 연결에서도 최대 패킷 수명 내에 순환하지 않습니다. 이 메커니즘은 더 빠른 링크에서 사용되면서 TCP에 필요해진 수정 사항입니다. RFC 1323에 기술되어 있으며 **PAWS** (**Protection Against Wrapped Sequence numbers**, 순환된 순서 번호에 대한 보호)라고 불립니다.

연결 간에, 초기 순서 번호에 대해, PAWS 이전에 TCP는 원래 위에서 설명한 시계 기반 방식을 사용했습니다. 그러나 이것이 보안 취약점(security vulnerability)이 있는 것으로 밝혀졌습니다. 시계 덕분에 공격자(attacker)가 다음 초기 순서 번호를 쉽게 예측하고 세 방향 핸드셰이크를 속이는 패킷을 보내 위조된 연결(forged connection)을 설정할 수 있었습니다. 이 허점을 막기 위해, 실제로는 **의사 난수(pseudorandom)**  초기 순서 번호가 사용됩니다. 그러나 관찰자에게 무작위로 보이더라도 초기 순서 번호가 일정 기간 동안 반복되지 않는 것이 여전히 중요합니다. 그렇지 않으면 지연된 중복이 혼란을 일으킬 수 있습니다.

## 6.2.3 연결 해제 (Connection Release)

연결을 해제하는 것은 설정하는 것보다 쉽습니다. 그럼에도 불구하고 예상보다 많은 함정이 있습니다. 앞서 언급했듯이, 연결을 종료하는 두 가지 방식이 있습니다: **비대칭 해제(asymmetric release)** 와 **대칭 해제(symmetric release)** .

### 비대칭 해제 (Asymmetric Release)

비대칭 해제는 전화 시스템의 방식과 같습니다: 한쪽이 전화를 끊으면 연결이 끊어집니다. 대칭 해제는 연결을 두 개의 별도 단방향 연결(unidirectional connections)로 취급하며 각각을 별도로 해제해야 합니다.

비대칭 해제는 갑작스럽고(abrupt) 데이터 손실(data loss)을 초래할 수 있습니다. 그림 6-12(Figure 6-12)의 시나리오를 살펴보세요.

<img width="400" height="368" alt="Image" src="https://github.com/user-attachments/assets/3155ec21-4608-45d6-bb9f-0fec32a98061" />

연결이 설정된 후 Host 1이 세그먼트를 보내고 Host 2에 정상적으로 도착합니다. 그런 다음 Host 1이 또 다른 세그먼트를 보냅니다. 불행히도 두 번째 세그먼트가 도착하기 전에 Host 2가 DISCONNECT를 발행합니다. 결과적으로 연결이 해제되고 데이터가 손실됩니다.

### 대칭 해제 (Symmetric Release)

데이터 손실을 피하기 위해 더 정교한 해제 프로토콜이 필요합니다. 한 가지 방법은 대칭 해제를 사용하는 것으로, 각 방향이 상대방과 독립적으로 해제됩니다. 여기서 호스트는 DISCONNECT 세그먼트를 보낸 후에도 계속 데이터를 수신할 수 있습니다.

대칭 해제는 각 프로세스가 보낼 고정된 양의 데이터가 있고 언제 보냈는지 명확히 아는 경우에 적합합니다. 다른 상황에서는 모든 작업이 완료되었고 연결을 종료해야 하는 시점을 결정하는 것이 그리 명확하지 않습니다. Host 1이 "나는 다 했어. 너도 다 했어?"라고 말하는 프로토콜을 생각할 수 있습니다. Host 2가 "나도 다 했어. 잘 가."라고 응답하면 연결이 안전하게 해제될 수 있습니다.

### 두 군대 문제 (Two-Army Problem)

불행히도 이 프로토콜은 항상 작동하지 않습니다. 이 문제를 설명하는 유명한 문제가 있으며, **두 군대 문제(two-army problem)** 라고 불립니다.

그림 6-13(Figure 6-13)에 표시된 것처럼, 백군(white army)이 계곡에 주둔해 있고, 양쪽 주변 언덕에는 청군(blue army)이 있다고 상상해 보세요. 백군은 어느 한 청군보다 크지만, 두 청군을 합치면 백군보다 큽니다. 한 청군이 단독으로 공격하면 패배하지만, 두 청군이 동시에 공격하면 승리합니다.

<img width="727" height="375" alt="Image" src="https://github.com/user-attachments/assets/330b6b99-45a6-415b-b758-ef2be1c23e2f" />

청군은 공격을 동기화하고 싶어합니다. 그러나 유일한 통신 수단은 계곡을 통해 도보로 전령을 보내는 것인데, 전령이 포로로 잡힐 수 있습니다(즉, 신뢰할 수 없는 통신 채널(unreliable communication channel)을 사용해야 합니다). 질문은: 청군이 이길 수 있는 프로토콜이 존재하는가입니다.

청군 #1의 사령관이 메시지를 보낸다고 가정합니다: "3월 29일 새벽에 공격하자. 어때?" 메시지가 도착하고 청군 #2의 사령관이 동의하며, 답장이 안전하게 청군 #1에게 전달된다고 가정합니다. 공격이 이루어질까요? 아마 아닐 것입니다. 사령관 #2는 자신의 답장이 전달되었는지 모르기 때문입니다. 전달되지 않았다면 청군 #1은 공격하지 않을 것이므로, 혼자 돌격하는 것은 어리석을 것입니다.

프로토콜을 세 방향 핸드셰이크(three-way handshake)로 개선해 봅시다. 최초 제안의 개시자가 응답을 확인해야 합니다. 메시지가 손실되지 않는다고 가정하면, 청군 #2는 확인을 받겠지만, 청군 #1의 사령관은 이제 망설일 것입니다. 결국 자신의 확인이 전달되었는지 모르며, 전달되지 않았다면 청군 #2가 공격하지 않을 것을 알기 때문입니다. 사방향 핸드셰이크(four-way handshake)를 만들 수도 있지만, 그것도 도움이 되지 않습니다.

사실, **작동하는 프로토콜은 존재하지 않음을 증명할 수 있습니다.** 어떤 프로토콜이 존재한다고 가정합니다. 프로토콜의 마지막 메시지가 필수적이거나 필수적이지 않습니다. 필수적이지 않다면, 그것(과 다른 비필수적 메시지)을 제거하여 모든 메시지가 필수적인 프로토콜을 남길 수 있습니다. 마지막 메시지가 전달되지 않으면 어떻게 될까요? 방금 그것이 필수적이라고 했으므로, 손실되면 공격이 이루어지지 않습니다. 마지막 메시지의 송신자는 도착을 확신할 수 없으므로 공격을 감행하지 않을 것입니다. 더 나쁜 것은, 다른 청군이 이를 알고 있으므로 역시 공격하지 않을 것입니다.

### 두 군대 문제와 연결 해제의 관련성

두 군대 문제가 연결 해제와 관련이 있음을 이해하려면, "공격(attack)"을 "연결 해제(disconnect)"로 대체하면 됩니다. 어느 쪽도 상대방이 연결 해제할 준비가 되었다고 확신할 때까지 연결 해제할 준비가 되지 않으면, 연결 해제는 결코 일어나지 않습니다.

### 실용적 해결: 세 방향 핸드셰이크와 타이머

실제로는 합의의 필요성을 포기하고 문제를 전송 사용자(transport user)에게 넘김으로써 이 곤경을 피할 수 있습니다. 각 측이 독립적으로 완료 시점을 결정하게 합니다. 이것이 더 쉬운 문제입니다.

그림 6-14(Figure 6-14)는 세 방향 핸드셰이크(three-way handshake)를 사용하여 해제하는 네 가지 시나리오를 보여줍니다. 이 프로토콜은 완벽하지 않지만 보통은 충분합니다.

<img width="755" height="813" alt="Image" src="https://github.com/user-attachments/assets/e7002f56-f784-4df3-a1c7-4ef5d5eedb74" />

#### 시나리오 (a): 정상 경우

그림 6-14(a)에서는 한 사용자가 연결 해제를 시작하기 위해 DR(DISCONNECTION REQUEST) 세그먼트를 보내는 정상 경우를 봅니다. DR이 도착하면, 수신자는 만약을 대비해 DR 세그먼트를 돌려보내고 타이머(timer)를 시작합니다. 이 DR이 도착하면, 원래 송신자가 ACK 세그먼트를 돌려보내고 연결을 해제합니다. 마지막으로 ACK 세그먼트가 도착하면 수신자도 연결을 해제합니다. 연결을 해제한다는 것은 전송 엔티티가 현재 열린 연결 테이블에서 해당 연결의 정보를 제거하고 연결 소유자(전송 사용자)에게 신호를 보내는 것을 의미합니다. 이 동작은 전송 사용자가 DISCONNECT 원시(primitive)를 발행하는 것과 다릅니다.

#### 시나리오 (b): 마지막 ACK 손실

그림 6-14(b)에 표시된 것처럼, 마지막 ACK 세그먼트가 손실되면 타이머에 의해 상황이 해결됩니다. 타이머가 만료되면 연결이 해제됩니다.

#### 시나리오 (c): 응답 DR 손실

이제 두 번째 DR이 손실되는 경우를 고려합니다. 연결 해제를 시작한 사용자는 예상 응답을 받지 못하고 타임아웃(time out)한 후 처음부터 다시 시작합니다. 그림 6-14(c)에서는 두 번째 시도에서 세그먼트가 손실되지 않고 모두 제시간에 전달되는 경우를 보여줍니다.

#### 시나리오 (d): 응답과 후속 DR 모두 손실

마지막 시나리오인 그림 6-14(d)는 그림 6-14(c)와 같지만 이번에는 반복된 재전송 시도에서 DR도 손실된 세그먼트로 인해 실패하는 경우입니다. **N**번 재시도 후 송신자는 포기하고 연결을 해제합니다. 한편 수신자도 타임아웃하고 역시 종료합니다.

> **그림 6-14 설명**: 연결 해제를 위한 네 가지 프로토콜 시나리오. (a) 세 방향 핸드셰이크의 정상 경우: Host 1이 DR을 보내고 타이머 시작, Host 2가 DR을 보내고 타이머 시작, Host 1이 연결 해제하고 ACK 보냄, Host 2가 ACK를 받고 연결 해제. (b) 마지막 ACK 손실: Host 1이 DR을 보냄, Host 2가 DR을 보내고 타이머 시작, Host 1이 연결 해제하고 ACK를 보내지만 ACK가 손실됨, Host 2는 타임아웃으로 연결 해제. (c) 응답 손실: Host 1이 DR을 보내고 타이머 시작, Host 2가 DR을 보내고 타이머 시작하지만 DR이 손실됨, Host 1이 타임아웃하여 DR 재전송하고 타이머 재시작, Host 2가 다시 DR을 보내고 타이머 시작, Host 1이 연결 해제하고 ACK 보냄, Host 2가 연결 해제. (d) 응답과 후속 DR 모두 손실: Host 1이 DR을 보내고 타이머 시작, Host 2가 DR을 보내고 타이머 시작하지만 DR이 손실됨, Host 1이 타임아웃하여 DR 재전송하지만 역시 손실됨, (N번 타임아웃) Host 1이 연결 해제, Host 2도 타임아웃으로 연결 해제.

이 프로토콜은 보통 충분하지만, 이론적으로 초기 DR과 N번의 재전송이 모두 손실되면 실패할 수 있습니다. 송신자는 포기하고 연결을 해제하지만, 상대방은 연결 해제 시도에 대해 전혀 모르며 여전히 완전히 활성 상태입니다. 이 상황은 **반개방 연결(half-open connection)** 을 초래합니다.

N번 재시도 후 송신자가 포기하지 않고 응답을 받을 때까지 영원히 계속하도록 하여 이 문제를 피할 수 있었습니다. 그러나 상대방이 타임아웃하도록 허용되면 송신자는 실제로 영원히 계속하게 될 것입니다. 응답이 영원히 오지 않을 것이기 때문입니다. 수신 측이 타임아웃하는 것을 허용하지 않으면 프로토콜은 그림 6-14(d)처럼 멈춰버립니다(hang).

### 반개방 연결 해결: 자동 연결 해제 규칙

반개방 연결을 처리하는 한 가지 방법은 일정 시간 동안 세그먼트가 도착하지 않으면 연결이 자동으로 해제되는 규칙을 두는 것입니다. 이렇게 하면 한 쪽이 연결 해제하면 상대방도 활동 부재를 감지하고 역시 연결 해제합니다. 이 규칙은 네트워크가 호스트 간에 더 이상 패킷을 전달할 수 없어 양쪽 모두 먼저 연결 해제하지 않은 상태에서 연결이 끊어지는 경우도 처리합니다.

물론 이 규칙이 도입되면 각 전송 엔티티에 세그먼트가 전송될 때마다 중지되고 재시작되는 타이머가 필요합니다. 이 타이머가 만료되면 상대방이 연결 해제하지 않도록 더미 세그먼트(dummy segment)가 전송됩니다. 반면에 자동 연결 해제 규칙이 사용되고 유휴 연결(idle connection)에서 너무 많은 더미 세그먼트가 연속으로 손실되면, 한 쪽이 먼저 자동 연결 해제하고 이어서 상대방도 자동 연결 해제합니다.

### TCP에서의 연결 해제

이 문제를 더 이상 논의하지는 않겠지만, 데이터 손실 없이 연결을 해제하는 것은 처음 보이는 것처럼 간단하지 않습니다. 여기서의 교훈은 **전송 사용자(transport user)가 언제 연결 해제할지 결정에 참여해야 한다**는 것입니다 -- 문제를 전송 엔티티만으로는 깔끔하게 해결할 수 없습니다.

애플리케이션의 중요성을 확인하려면, TCP가 일반적으로 대칭적 종료(symmetric close)를 수행하는 것을 고려해 보세요(각 측이 FIN 패킷으로 자신의 절반 연결을 독립적으로 닫음). 그러나 많은 웹 서버는 클라이언트에게 RST 패킷을 보내 갑작스럽게(abrupt) 연결을 종료합니다. 이것이 작동하는 이유는 웹 서버가 데이터 교환 패턴을 알기 때문입니다. 먼저 클라이언트로부터 요청(클라이언트가 보낼 모든 데이터)을 수신하고, 그 다음 클라이언트에게 응답을 보냅니다. 웹 서버가 응답을 완료하면 양방향 모두에서 모든 데이터가 전송된 것입니다. 서버는 클라이언트에게 경고를 보내고 갑작스럽게 연결을 닫을 수 있습니다. 클라이언트가 이 경고를 받으면 연결 상태를 즉시 해제합니다. 클라이언트가 경고를 받지 못하면 결국 서버가 더 이상 통신하지 않는 것을 알게 되고 연결 상태를 해제합니다. 어느 경우든 데이터는 성공적으로 전달된 것입니다.

## 6.2.4 오류 제어와 흐름 제어 (Error Control and Flow Control)

연결 설정과 해제를 살펴보았으니, 이제 연결이 사용 중일 때 어떻게 관리되는지 살펴보겠습니다. 핵심 이슈는 **오류 제어(error control)** 와 **흐름 제어(flow control)** 입니다. 오류 제어는 데이터가 원하는 수준의 신뢰성(reliability)으로 전달되도록 보장하는 것으로, 일반적으로 모든 데이터가 오류 없이 전달되는 것을 의미합니다. 흐름 제어는 빠른 송신자(fast transmitter)가 느린 수신자(slow receiver)를 압도하지 않도록 하는 것입니다.

### 데이터 링크 계층과의 유사점

이 두 이슈 모두 데이터 링크 계층을 공부할 때 이미 접한 적이 있습니다. 전송 계층에서 사용되는 해결책은 3장에서 공부한 것과 동일한 메커니즘입니다. 매우 간략한 요약으로:

1. 프레임이 **오류 검출 코드(error-detecting code)** (예: CRC 또는 체크섬(checksum))를 운반하며, 정보가 올바르게 수신되었는지 확인하는 데 사용됩니다.

2. 프레임이 자신을 식별하는 **순서 번호(sequence number)** 를 운반하며, 수신자로부터 성공적 수신의 확인 응답을 받을 때까지 송신자에 의해 재전송됩니다. 이것을 **ARQ** (**Automatic Repeat reQuest**, 자동 반복 요청)라고 합니다.

3. 송신자가 미해결 상태로 유지할 수 있는 프레임의 최대 수가 있으며, 수신자가 프레임을 충분히 빠르게 확인하지 않으면 일시 중지(pause)합니다. 이 최대값이 하나의 패킷이면 프로토콜은 **정지-대기(stop-and-wait)** 라고 합니다. 더 큰 윈도우는 파이프라이닝(pipelining)을 가능하게 하고 긴 고속 링크에서 성능을 향상시킵니다.

4. **슬라이딩 윈도우(sliding window)**  프로토콜이 이러한 기능들을 결합하며 양방향 데이터 전송(bidirectional data transfer)을 지원하는 데에도 사용됩니다.

### 링크 계층과 전송 계층의 차이점

이러한 메커니즘이 링크 계층의 프레임에 사용된다면, 전송 계층의 세그먼트에도 사용될 것이라고 자연스럽게 생각할 수 있습니다. 그러나 실제로는 링크 계층과 전송 계층 사이에 중복이 거의 없습니다. 동일한 메커니즘이 사용되더라도 기능과 정도에서 차이가 있습니다.

#### 기능의 차이: 오류 검출

기능의 차이로 오류 검출을 고려해 보겠습니다. **링크 계층 체크섬(link layer checksum)** 은 단일 링크(single link)를 건너는 프레임을 보호합니다. **전송 계층 체크섬(transport layer checksum)** 은 전체 네트워크 경로(entire network path)를 건너는 세그먼트를 보호합니다. 이것은 **종단 간 검사(end-to-end check)** 이며, 모든 링크에 대한 검사를 갖는 것과 같지 않습니다.

Saltzer 등(1984)은 라우터 내부에서 패킷이 손상되는 상황을 기술했습니다. 링크 계층 체크섬은 패킷이 링크를 건너는 동안만 보호했지, 라우터 내부에 있을 때는 보호하지 않았습니다. 따라서 패킷이 모든 링크의 검사에 따르면 올바르더라도 잘못 전달되었습니다.

이 예와 다른 예들이 Saltzer 등으로 하여금 **종단 간 논증(end-to-end argument)** 을 명확히 하게 했습니다. 이 논증에 따르면, 종단 간으로 실행되는 전송 계층 검사가 정확성을 위해 필수적(essential)이며, 링크 계층 검사는 필수적이지는 않지만 성능 향상에는 가치가 있습니다(이것이 없으면 손상된 패킷이 전체 경로를 따라 불필요하게 전송될 수 있으므로).

#### 정도의 차이: 재전송과 슬라이딩 윈도우

정도의 차이로 재전송과 슬라이딩 윈도우를 고려해 보겠습니다. 위성 링크가 아닌 대부분의 무선 링크는 한 번에 하나의 프레임만 미해결 상태로 둘 수 있습니다. 즉, 링크의 대역폭-지연 곱(bandwidth-delay product)이 작아서 전체 프레임도 링크 내에 저장할 수 없습니다. 이 경우 작은 윈도우 크기로 충분히 좋은 성능을 얻을 수 있습니다. 예를 들어, 802.11은 정지-대기(stop-and-wait) 프로토콜을 사용하여 각 프레임을 전송하거나 재전송하고 다음 프레임으로 넘어가기 전에 확인을 기다립니다. 하나보다 큰 윈도우 크기는 복잡성만 추가하고 성능 향상에는 기여하지 않습니다. (스위칭된) 이더넷이나 ISP 백본과 같은 유선 및 광섬유 링크의 경우 오류율이 충분히 낮아서 링크 계층 재전송을 생략해도 됩니다. 종단 간 재전송이 잔여 프레임 손실(residual frame loss)을 복구할 것이기 때문입니다.

반면에 많은 TCP 연결은 단일 세그먼트보다 훨씬 큰 **대역폭-지연 곱(bandwidth-delay product)** 을 가지고 있습니다. 미국 횡단 1 Mbps 연결에서 왕복 시간(round-trip time)이 100밀리초(msec)인 경우를 생각해 보세요. 이 느린 연결에서도 세그먼트를 보내고 확인 응답을 받는 시간 동안 200 Kbit의 데이터가 수신자에 저장됩니다. 이러한 상황에서는 큰 슬라이딩 윈도우가 사용되어야 합니다. 정지-대기는 성능을 200 msec마다 하나의 세그먼트, 즉 네트워크가 실제로 아무리 빨라도 초당 5세그먼트로 제한하게 됩니다.

### 버퍼링 전략

전송 프로토콜이 일반적으로 더 큰 슬라이딩 윈도우를 사용한다는 점을 감안하여, 데이터 버퍼링(buffering) 문제를 더 자세히 살펴보겠습니다. 호스트는 많은 연결을 가질 수 있으며, 각 연결은 별도로 취급되므로 슬라이딩 윈도우를 위한 상당한 양의 버퍼링이 필요할 수 있습니다. 버퍼는 송신자와 수신자 양쪽에 필요합니다. 확실히 송신자에는 모든 전송되었지만 아직 확인되지 않은 세그먼트를 보관하는 버퍼가 필요합니다. 이 세그먼트들이 손실되어 재전송해야 할 수 있기 때문입니다.

#### 송신자 측 버퍼링

송신자가 버퍼링을 하므로, 수신자는 특정 연결에 버퍼를 할당할 수도 있고 하지 않을 수도 있습니다. 수신자는 예를 들어 모든 연결이 공유하는 단일 버퍼 풀(buffer pool)을 유지할 수 있습니다. 세그먼트가 들어오면 새 버퍼를 동적으로 획득하려 합니다. 사용 가능하면 세그먼트가 수락되고, 그렇지 않으면 폐기됩니다. 송신자는 네트워크에 의해 손실된 세그먼트를 재전송할 준비가 되어 있으므로, 수신자가 세그먼트를 드롭해도 영구적인 피해는 없습니다. 일부 자원이 낭비되긴 하지만 말입니다. 송신자는 확인 응답을 받을 때까지 계속 시도합니다.

#### 출발지 버퍼링과 목적지 버퍼링의 절충

출발지 버퍼링(source buffering)과 목적지 버퍼링(destination buffering) 사이의 최적 절충은 연결이 운반하는 트래픽 유형에 따라 다릅니다. 대화형 터미널(interactive terminal)이 생성하는 것과 같은 저대역폭 버스트 트래픽(low-bandwidth bursty traffic)의 경우, 어느 쪽에도 버퍼를 할당하지 않고 양쪽에서 동적으로 획득하되, 세그먼트가 가끔 폐기되어야 하는 경우 송신자에서 버퍼링하는 것이 합리적입니다. 반면에 파일 전송 및 기타 고대역폭 트래픽(high-bandwidth traffic)의 경우, 수신자가 데이터를 최대 속도로 흐르게 하기 위해 전체 윈도우만큼의 버퍼를 할당하는 것이 더 좋습니다. 이것이 TCP가 사용하는 전략입니다.

버퍼 풀을 어떻게 구성할 것인가의 문제가 여전히 남아 있습니다.

> **그림 6-15 설명**: (a) 체인형 고정 크기 버퍼(Chained fixed-size buffers): 동일한 크기의 버퍼들이 연결 리스트로 연결되어 있으며, 각 버퍼에 세그먼트 하나씩 저장됩니다. (b) 체인형 가변 크기 버퍼(Chained variable-size buffers): 서로 다른 크기의 버퍼들이 연결 리스트로 연결되어 있으며, 회색으로 표시된 부분이 사용된 공간입니다. (c) 연결당 하나의 큰 순환 버퍼(One large circular buffer per connection): 하나의 큰 버퍼에 여러 세그먼트(Segment 1, Segment 2, Segment 3, Segment 4)가 저장되어 있으며, 미사용 공간(Unused space)도 있습니다.

대부분의 세그먼트가 거의 같은 크기라면, 버퍼당 하나의 세그먼트로 동일 크기의 버퍼 풀을 구성하는 것이 자연스럽습니다(그림 6-15(a)). 그러나 웹 페이지의 짧은 요청에서 피어 투 피어 파일 전송의 큰 패킷에 이르기까지 세그먼트 크기에 넓은 변동이 있는 경우, 고정 크기 버퍼 풀은 문제를 일으킵니다. 버퍼 크기가 가능한 가장 큰 세그먼트와 같다면, 짧은 세그먼트가 도착할 때마다 공간이 낭비됩니다. 버퍼 크기가 최대 세그먼트 크기보다 작다면, 긴 세그먼트에는 여러 버퍼가 필요하며 부수적인 복잡성이 수반됩니다.

버퍼 크기 문제의 또 다른 접근법은 가변 크기 버퍼(variable-sized buffers)를 사용하는 것입니다(그림 6-15(b)). 장점은 더 나은 메모리 활용이지만, 더 복잡한 버퍼 관리가 필요합니다. 세 번째 가능성은 연결당 하나의 큰 순환 버퍼(circular buffer)를 할당하는 것입니다(그림 6-15(c)). 이 시스템은 단순하고 우아하며 세그먼트 크기에 의존하지 않지만, 연결이 많이 사용될 때만 메모리를 효율적으로 사용합니다.

### 동적 버퍼 할당 (Dynamic Buffer Allocation)

연결이 열리고 닫히면서 트래픽 패턴이 변하면, 송신자와 수신자는 버퍼 할당을 동적으로 조정해야 합니다. 따라서 전송 프로토콜은 송신 호스트가 상대방의 버퍼 공간을 요청할 수 있도록 해야 합니다. 버퍼는 연결별(per connection) 또는 두 호스트 간의 모든 연결에 대해 집합적(collectively)으로 할당될 수 있습니다. 대안적으로, 수신자가 버퍼 상황을 알고 있지만 제공되는 트래픽은 모르는 상태에서, 송신자에게 "X개의 버퍼를 예약해 두었습니다"라고 알릴 수 있습니다. 열린 연결 수가 증가하면 할당을 줄여야 할 수도 있으므로, 프로토콜은 이 가능성에 대비해야 합니다.

동적 버퍼 할당을 관리하는 합리적으로 일반적인 방법은 3장의 슬라이딩 윈도우 프로토콜과 대조적으로 버퍼링을 확인 응답에서 분리(decouple)하는 것입니다. 동적 버퍼 관리는 실질적으로 가변 크기 윈도우를 의미합니다. 처음에 송신자는 예상 필요에 따라 일정 수의 버퍼를 요청합니다. 수신자는 감당할 수 있는 만큼을 승인합니다. 송신자가 세그먼트를 전송할 때마다 할당량을 감소시키고, 할당량이 0에 도달하면 완전히 중지합니다. 수신자는 확인 응답과 버퍼 할당을 역방향 트래픽에 별도로 피기백(piggyback)합니다. TCP는 이 방식을 사용하며, **윈도우 크기(Window size)** 라는 헤더 필드에 버퍼 할당을 운반합니다.

### 버퍼 할당 세그먼트 손실 시의 교착 상태 문제

이러한 종류의 버퍼 할당 방식에서 데이터그램 네트워크의 제어 세그먼트(control segment)가 손실될 수 있으며 -- 실제로 그럴 가능성이 높은 -- 문제가 발생할 수 있습니다. 16번 줄을 보세요. B가 이제 A에 더 많은 버퍼를 할당했지만 할당 세그먼트가 손실되었습니다. 제어 세그먼트가 순서화(sequenced)되거나 타임아웃되지 않으므로, A는 이제 **교착 상태(deadlocked)** 입니다. 이 상황을 방지하기 위해, 각 호스트는 주기적으로(periodically) 각 연결에 대한 확인 응답과 버퍼 상태를 제공하는 제어 세그먼트를 보내야 합니다. 이렇게 하면 교착 상태가 조만간 해소될 것입니다.

### 네트워크 운반 용량 제한

지금까지 송신자의 데이터 전송률에 부과되는 유일한 제한이 수신자의 가용 버퍼 공간 양이라고 암묵적으로 가정했습니다. 이것이 항상 해당되는 것은 아닙니다. 메모리 가격이 한때는 비쌌지만 극적으로 떨어졌습니다. 호스트가 충분한 메모리를 갖추어 버퍼 부족이 문제가 되는 경우는 드뭅니다. 물론 이것은 버퍼 크기가 충분히 크게 설정되어야 하는 것에 달려 있으며, TCP의 경우 항상 그렇지는 않았습니다(Zhang 등, 2002).

버퍼 공간이 더 이상 최대 흐름을 제한하지 않으면, 또 다른 병목(bottleneck)이 나타납니다: **네트워크의 운반 용량(carrying capacity)** . 인접 라우터가 최대 초당 **x** 패킷을 교환할 수 있고 한 쌍의 호스트 사이에 **k**개의 분리된 경로(disjoint path)가 있다면, 양쪽 끝에 아무리 많은 버퍼 공간이 있더라도 초당 **kx** 세그먼트 이상을 교환할 방법이 없습니다. 송신자가 너무 세게 밀면(즉, 초당 kx 세그먼트 이상을 보내면) 네트워크가 혼잡(congested)해져서 들어오는 만큼 빠르게 세그먼트를 전달할 수 없게 됩니다.

### 동적 윈도우와 네트워크 용량

필요한 것은 수신자의 버퍼링 용량이 아닌 네트워크의 운반 용량에 기반하여 송신자의 전송을 제한하는 메커니즘입니다. Belsnes(1975)는 송신자가 네트워크의 운반 용량에 맞추어 윈도우 크기를 동적으로 조정하는 슬라이딩 윈도우 흐름 제어 방식을 제안했습니다. 이는 동적 슬라이딩 윈도우가 흐름 제어(flow control)와 혼잡 제어(congestion control)를 모두 구현할 수 있음을 의미합니다.

네트워크가 초당 **c** 세그먼트를 처리할 수 있고, 왕복 시간(round-trip time, 전송, 전파, 수신자에서의 큐잉, 처리, 확인 응답 반환 포함)이 **r**이면, 송신자의 윈도우는 **cr**이어야 합니다. 이 크기의 윈도우로 송신자는 일반적으로 파이프라인을 가득 채운 상태로 동작합니다. 네트워크 성능이 조금만 감소해도 차단(block)이 발생합니다. 네트워크에서 주어진 흐름에 사용 가능한 용량은 시간에 따라 변하므로, 운반 용량의 변화를 추적하기 위해 윈도우 크기를 자주 조정해야 합니다. 나중에 보겠지만, TCP는 유사한 방식을 사용합니다.

## 6.2.5 다중화 (Multiplexing)

**다중화(multiplexing)** , 즉 연결, 가상 회선(virtual circuit), 물리적 링크를 통한 여러 대화의 공유는 네트워크 아키텍처의 여러 계층에서 역할을 합니다. 전송 계층에서 다중화의 필요성은 여러 가지 방식으로 발생할 수 있습니다.

### 상향 다중화 (Upward Multiplexing)

예를 들어, 호스트에 하나의 네트워크 주소(network address)만 사용 가능한 경우, 해당 머신의 모든 전송 연결이 그것을 사용해야 합니다. 세그먼트가 들어오면 어떤 프로세스에게 전달할지 결정할 방법이 필요합니다. 이 상황을 **다중화(multiplexing)** 라고 하며 그림 6-17(a)(Figure 6-17(a))에 나타나 있습니다.

<img width="712" height="394" alt="Image" src="https://github.com/user-attachments/assets/bb163932-0798-44a6-b621-aaf98f413523" />

> **그림 6-17 설명**: (a) 다중화(Multiplexing): 여러 전송 주소(Transport address)에서 온 네 개의 서로 다른 전송 연결이 모두 동일한 네트워크 연결(Network address)을 사용하여 원격 호스트로 갑니다. 계층이 표시됩니다(Layer 4 = 전송, Layer 3 = 네트워크, Layer 2 = 데이터 링크, Layer 1 = 물리). 라우터 회선(Router lines)이 Layer 2에서 아래로 라우터로 향합니다. 이 그림에서 네 개의 서로 다른 전송 연결이 모두 동일한 네트워크 연결(예: IP 주소)을 사용하여 원격 호스트에 접속합니다.

### 역다중화 (Inverse Multiplexing)

다중화는 전송 계층에서 또 다른 이유로 유용할 수 있습니다. 사용자가 하나의 네트워크 경로가 제공할 수 있는 것보다 더 많은 대역폭이나 더 높은 신뢰성이 필요하다고 가정합니다. 이때 해결책은 여러 네트워크 경로에 트래픽을 라운드 로빈(round-robin) 방식으로 분배하는 연결을 갖는 것입니다. 이것은 그림 6-17(b)에 나타나 있습니다. 이 방식은 **역다중화(inverse multiplexing)** 라고 불립니다. **k**개의 네트워크 연결이 열려 있으면 유효 대역폭이 **k**배 증가할 수 있습니다.

역다중화의 예로는 **SCTP** (**Stream Control Transmission Protocol**, 스트림 제어 전송 프로토콜)가 있으며, 여러 네트워크 인터페이스를 사용하여 연결을 실행할 수 있습니다. 이와 대조적으로 TCP는 단일 네트워크 종단점(endpoint)을 사용합니다. 역다중화는 여러 저속 링크가 하나의 고속 링크로 병렬로 사용될 때 링크 계층에서도 발견됩니다.

## 6.2.6 충돌 복구 (Crash Recovery)

호스트와 라우터가 충돌(crash)할 수 있거나 연결이 장기간(예: 대용량 소프트웨어나 미디어 다운로드) 지속되는 경우, 이러한 충돌로부터의 복구(recovery)가 중요한 문제가 됩니다.

### 라우터 충돌 vs 호스트 충돌

전송 엔티티가 전적으로 호스트 내에 있다면, 네트워크와 라우터의 충돌로부터의 복구는 간단합니다. 전송 엔티티는 항상 세그먼트가 손실될 것을 예상하고 재전송으로 대처하는 방법을 알고 있습니다.

더 골치 아픈 문제는 **호스트 충돌(host crash)** 로부터 복구하는 방법입니다. 특히 서버가 충돌하고 빠르게 재부팅할 때 클라이언트가 계속 작업할 수 있는 것이 바람직합니다.

### 호스트 충돌 복구의 어려움

어려움을 설명하기 위해, 한 호스트(클라이언트)가 다른 호스트(파일 서버)에 긴 파일을 보내고 있다고 가정합니다. 간단한 정지-대기(stop-and-wait) 프로토콜을 사용합니다. 서버의 전송 계층은 들어오는 세그먼트를 전송 사용자에게 하나씩 전달합니다. 전송 도중 서버가 충돌합니다. 다시 시작하면 테이블이 재초기화(reinitialized)되어 정확히 어디에 있었는지 더 이상 알 수 없습니다.

### 복구 시도: 브로드캐스트 방식

이전 상태를 복구하려는 시도로, 서버가 모든 다른 호스트에 브로드캐스트 세그먼트(broadcast segment)를 보내 방금 충돌했으며 클라이언트가 모든 열린 연결의 상태를 알려달라고 요청할 수 있습니다.

각 클라이언트는 두 상태 중 하나에 있을 수 있습니다: 하나의 세그먼트가 미해결(**S1**, one segment outstanding), 또는 미해결 세그먼트 없음(**S0**, no segments outstanding). 이 상태 정보만으로 클라이언트는 가장 최근 세그먼트를 재전송할지 여부를 결정해야 합니다.

### 단순한 접근의 문제점

언뜻 보면 명백해 보입니다: 클라이언트는 충돌 소식을 들었을 때 미확인 세그먼트가 있는 경우(즉, 상태 **S1**인 경우)에만 재전송해야 합니다. 그러나 더 자세히 살펴보면 이 순진한 접근에 어려움이 드러납니다.

예를 들어, 서버의 전송 엔티티가 먼저 확인 응답(A)을 보내고, 확인 응답이 전송된 후에 출력 스트림(output stream)에 쓰기(W)를 하는 상황을 고려합니다. 충돌이 확인 응답은 보냈지만 쓰기가 완전히 완료되기 전에 발생하면, 클라이언트는 확인 응답을 받고 충돌 복구 알림이 도착했을 때 상태 **S0**에 있게 됩니다. 클라이언트는 따라서 재전송하지 않으며, (잘못되게) 세그먼트가 도착했다고 판단합니다. 이 결정은 세그먼트가 누락되는 결과를 초래합니다.

이 시점에서 "그 문제는 쉽게 해결할 수 있다. 먼저 쓰기를 하고 그 다음에 확인 응답을 보내도록 전송 엔티티를 재프로그래밍하면 된다"고 생각할 수 있습니다. 다시 시도해 봅시다. 쓰기는 완료되었지만 확인 응답이 전송되기 전에 충돌이 발생한다고 상상합니다. 클라이언트는 상태 **S1**에 있게 되어 재전송하며, 서버 애플리케이션 프로세스의 출력 스트림에 감지되지 않는 중복 세그먼트(undetected duplicate)가 들어가게 됩니다.

### 클라이언트와 서버 전략의 조합: 그림 6-18

클라이언트와 서버가 어떻게 프로그래밍되든, 프로토콜이 올바르게 복구하지 못하는 상황이 항상 존재합니다. 서버는 두 가지 방식 중 하나로 프로그래밍될 수 있습니다: **먼저 확인 후 쓰기(acknowledge first, then write)**  또는 **먼저 쓰기 후 확인(write first, then acknowledge)** . 클라이언트는 네 가지 방식으로 프로그래밍될 수 있습니다: **항상 재전송(always retransmit)** , **재전송 안 함(never retransmit)** , **S0일 때만 재전송(retransmit only in S0)** , 또는 **S1일 때만 재전송(retransmit only in S1)** . 이는 8가지 조합을 제공하지만, 곧 보게 되듯이 각 조합에 대해 프로토콜을 실패하게 만드는 이벤트 시퀀스가 존재합니다.

서버에서 세 가지 이벤트가 가능합니다: 확인 응답 전송(**A**), 출력 프로세스에 쓰기(**W**), 그리고 충돌(**C**). 세 이벤트는 여섯 가지 순서로 발생할 수 있습니다: **AC(W)** , **AWC**, **C(AW)** , **C(WA)** , **WAC**, **WC(A)** . 여기서 괄호는 A도 W도 C를 따를 수 없음을 나타내는 데 사용됩니다(즉, 한번 충돌하면 충돌한 것입니다).

그림 6-18(Figure 6-18)은 8가지 클라이언트-서버 전략 조합과 각각에 대한 유효한 이벤트 시퀀스를 모두 보여줍니다.

> **그림 6-18 설명**: 클라이언트와 서버 전략의 다양한 조합. 테이블은 두 부분으로 나뉩니다: 수신 호스트(서버)가 사용하는 전략으로 "먼저 ACK 후 쓰기(First ACK, then write)"와 "먼저 쓰기 후 ACK(First write, then ACK)"가 있습니다.

범례:
- **OK** = 프로토콜이 올바르게 기능함
- **DUP** = 프로토콜이 중복 메시지를 생성함
- **LOST** = 프로토콜이 메시지를 손실함

예를 들어, 클라이언트가 항상 재전송하면, **AWC** 이벤트는 감지되지 않는 중복을 생성합니다. 다른 두 이벤트가 올바르게 작동하더라도 말입니다.

**각 전략에 대해 프로토콜을 실패하게 만드는 이벤트 시퀀스가 존재한다는 것에 주목하세요.** 송신자가 ACK를 받았는지 여부에 따라 결정을 내리고, 수신자가 쓰기와 확인 응답을 별도의 불가분(atomic)이 아닌 동작으로 수행하는 한, 모든 충돌에서 올바르게 복구하는 것은 불가능합니다. 전송 계층은 호스트 충돌로부터 완전히 투명하게 복구할 수 없으므로, 복구는 상위 계층에서 처리되어야 합니다.

보다 일반적으로 말하면, 이 결과는 충돌 복구가 상위 수준 프로토콜에서만 수행될 수 있다는 것을 보여주며, 이는 또한 종단 간 논증(end-to-end argument)의 적용 범위에 속합니다.
