# 3.4 슬라이딩 윈도우 프로토콜 (Sliding Window Protocols)

### 도입부: 양방향 전송과 피기백킹의 필요성

대부분의 실제 상황에서는 **양방향(full-duplex)** 으로 데이터를 전송해야 합니다. 더 나은 방법은 **동일한 링크를 양방향으로 사용**하는 것입니다.

데이터 프레임이 도착했을 때 즉시 별도의 제어 프레임을 보내는 대신, 수신자는 확인응답을 나가는 데이터 프레임에 첨부합니다. **나가는 확인응답을 일시적으로 지연시켜 다음 나가는 데이터 프레임에 편승시키는 기법을 피기백킹(piggybacking)** 이라고 합니다.

#### 피기백킹의 장점

- **가용 채널 대역폭을 더 잘 활용**할 수 있습니다.
- 프레임 수가 적으면 수신자의 처리 부하도 줄어듭니다.

#### 피기백킹의 문제점

확인응답을 편승시킬 패킷을 **얼마나 오래 기다려야 하는가**의 문제입니다. 너무 오래 기다리면 송신자의 타임아웃이 발생합니다. 실제로는 일정 고정 시간을 기다리며, 새 패킷이 빨리 도착하면 편승시키고, 기간이 끝나면 별도의 확인응답 프레임을 보냅니다.

### 슬라이딩 윈도우 프로토콜의 핵심 개념

#### 순서 번호 (Sequence Numbers)

모든 슬라이딩 윈도우 프로토콜에서 각 나가는 프레임에는 순서 번호가 포함되며, 0부터 어떤 최댓값까지의 범위를 갖습니다. 최댓값은 보통 **2^n - 1**이므로 순서 번호는 정확히 *n*-비트 필드에 들어갑니다.

#### 송신 윈도우 (Sending Window)

**어느 시점에서든 송신자가 전송이 허용된 프레임에 해당하는 순서 번호의 집합**을 유지합니다. 새 패킷이 도착하면 윈도우의 상단 경계가 전진하고, 확인응답이 도착하면 하단 경계가 전진합니다.

송신자의 윈도우 내 프레임들은 전송 중 손실되거나 손상될 수 있으므로, 송신자는 가능한 재전송을 위해 이 프레임들을 모두 메모리에 보관해야 합니다. **최대 윈도우 크기가 *n*이면 송신자는 *n*개의 버퍼가 필요합니다.**

#### 수신 윈도우 (Receiving Window)

수신자도 수락이 허용된 프레임의 집합에 해당하는 **수신 윈도우(receiving window)** 를 유지합니다. **윈도우 크기가 1**이면 순서대로만 프레임을 수락합니다.

### 3.4.1 1비트 슬라이딩 윈도우 프로토콜 (A One-Bit Sliding Window Protocol)

**윈도우 크기가 1**인 슬라이딩 윈도우 프로토콜로, 정지-대기(stop-and-wait) 방식을 사용합니다. 프로토콜의 송신자와 수신자가 하나의 프로시저에 통합되어 **양방향(bidirectional)**  통신을 지원합니다.

순서 번호는 0과 1만 가능합니다. 확인응답 필드에는 마지막으로 오류 없이 수신된 프레임의 번호가 포함됩니다.

#### 동기화 문제

**양쪽이 동시에 초기 프레임을 전송하는 경우** 특이한 상황이 발생합니다:

- **정상적 경우**: B가 A의 첫 프레임을 기다린 후 자신의 프레임을 보냅니다. 각 프레임 도착마다 새 패킷이 전달됩니다.
- **비정상적 경우**: A와 B가 동시에 시작하면 **프레임의 절반이 중복**을 포함합니다. 전송 오류가 없음에도 대역폭이 낭비됩니다.

그럼에도 프레임 분실이나 조기 타임아웃의 어떤 조합도 이 프로토콜이 중복 패킷을 전달하거나 교착 상태에 빠지게 할 수 없습니다. **프로토콜은 올바릅니다**.

### 3.4.2 Go-Back-N을 사용하는 프로토콜 (A Protocol Using Go-Back-N)

#### 파이프라이닝의 필요성

긴 전파 지연이 있는 채널에서 정지-대기 프로토콜은 매우 비효율적입니다.

**위성 채널 예시**: 500밀리초 왕복 전파 지연, 50kbps 채널, 1000비트 프레임의 경우:
- 프레임 전송 시간: 20msec
- 확인응답 도착까지: 520msec
- **가용 대역폭의 4%만 사용**

#### 대역폭-지연 곱 (Bandwidth-Delay Product)

적절한 윈도우 크기 *w*는 다음으로 결정됩니다:

> **w = 2BD + 1** (BD = 대역폭 × 편도 전송 시간 / 프레임 크기)

위 예시에서: BD = 12.5 프레임, w = **26 프레임**

#### 링크 이용률 공식

> **링크 이용률 ≤ w / (1 + 2BD)** 

#### 파이프라이닝 (Pipelining)

**여러 프레임을 동시에 전송 중인 상태로 유지하는 기법**입니다. 중간의 프레임이 손상되거나 분실되면, 후속 프레임 처리에 대해 두 가지 접근법이 있습니다:

#### 접근법 1: Go-Back-N

- 수신자가 폐기된 프레임 이후의 **모든 후속 프레임을 폐기**합니다.
- **수신 윈도우 크기 1**에 해당합니다.
- 송신자는 타이머 만료 시 손상된 프레임부터 **모든 미확인 프레임을 재전송**합니다.
- 오류율이 높으면 많은 대역폭을 낭비할 수 있습니다.

#### 접근법 2: Selective Repeat

- 수신된 나쁜 프레임은 폐기하지만, 그 뒤의 좋은 프레임은 **수락하고 버퍼링**합니다.
- 송신자는 **가장 오래된 미확인 프레임만 재전송**합니다.
- 1보다 큰 수신자 윈도우에 해당합니다.
- 대량의 메모리를 필요로 할 수 있습니다.

두 접근법은 **대역폭**과 **데이터 링크 계층 버퍼 공간** 사이의 트레이드오프입니다.

#### Go-Back-N 프로토콜 상세 (프로토콜 5)

**미처리 프레임의 최대 수와 순서 번호 공간:**

MAX_SEQ + 1개의 서로 다른 순서 번호가 있지만, **MAX_SEQ개의 프레임**만 미처리될 수 있습니다. 이 제한이 필요한 이유: 송신자가 전체 윈도우를 전송하고 모든 확인응답이 도착한 후 다시 같은 번호의 프레임을 보내면, 수신자가 새 배치인지 재전송인지 **구별할 수 없기** 때문입니다.

**누적 확인응답 (Cumulative Acknowledgement):**

프레임 *n*에 대한 확인응답이 도착하면 프레임 *n-1*, *n-2* 등도 자동으로 확인응답됩니다. 이는 이전 확인응답이 분실된 경우에 특히 중요합니다.

**다중 타이머:**

프레임마다 독립적인 타이머가 필요합니다. 대기 중인 타임아웃은 연결 리스트로 관리되며, 각 노드는 만료까지의 클럭 틱 수, 타이밍 대상 프레임, 다음 노드 포인터를 포함합니다.

### 3.4.3 선택적 재전송을 사용하는 프로토콜 (A Protocol Using Selective Repeat)

#### 선택적 재전송의 동기

Go-Back-N은 오류가 드문 경우 잘 작동하지만, 회선 품질이 나쁘면 **재전송되는 프레임에 대해 많은 대역폭을 낭비**합니다.

#### 윈도우 크기 제한 문제

비순차적 수신은 프레임 순서 번호에 대한 더 엄격한 제약을 도입합니다.

**문제 시나리오**: 3비트 순서 번호(0~7)에서 최대 7개의 프레임을 미처리 허용할 경우:
- 모든 프레임이 올바르게 수신되고 확인응답이 모두 손실되면
- 수신자 윈도우가 전진하여 새로운 유효 범위와 이전 범위가 **겹칩니다**
- 재전송된 프레임을 새 프레임으로 착각하여 **프로토콜이 실패**합니다

**해결책**: **최대 윈도우 크기가 순서 번호 범위의 최대 절반 이하**여야 합니다.

일반적으로 선택적 재전송 프로토콜에서 윈도우 크기는 **(MAX_SEQ + 1) / 2**가 됩니다.

#### 버퍼와 타이머의 수

필요한 버퍼의 수는 **윈도우 크기와 같으며, 순서 번호 공간의 크기가 아닙니다**. 같은 이유로 필요한 타이머의 수도 버퍼의 수와 같습니다.

#### 보조 타이머 (Auxiliary Timer)와 단방향 트래픽 처리

역방향 트래픽이 적으면 확인응답을 피기백할 프레임이 없어 문제가 됩니다. **보조 타이머(auxiliary timer)** 가 시작되고, 만료 전에 역방향 트래픽이 없으면 **별도의 확인응답 프레임**이 전송됩니다.

**중요**: 보조 타이머의 타임아웃은 데이터 프레임의 타임아웃보다 **상당히 짧아야** 합니다.

#### NAK (부정 확인응답) 처리

수신자가 오류를 의심할 때 **부정 확인응답(NAK)**  프레임을 보냅니다:
1. **손상된 프레임이 도착할 때**
2. **기대한 것과 다른 프레임이 도착할 때** (잠재적 프레임 분실)

같은 프레임에 대한 중복 NAK 방지를 위해 `no_nak` 변수로 추적합니다.

#### Go-Back-N과 Selective Repeat의 핵심 비교

| 특성 | Go-Back-N | Selective Repeat |
|------|-----------|-----------------|
| **수신 윈도우 크기** | 1 | (MAX_SEQ + 1) / 2 |
| **오류 발생 시 동작** | 오류 프레임 이후 모든 프레임 폐기, 전부 재전송 | 오류 프레임만 재전송, 나머지는 버퍼링 |
| **최대 미처리 프레임 수** | MAX_SEQ | (MAX_SEQ + 1) / 2 |
| **수신자 버퍼** | 1개 | 윈도우 크기만큼 |
| **NAK 사용** | 사용하지 않음 | 사용 (선택적) |
| **보조 타이머** | 없음 | 있음 |
| **대역폭 효율 (오류 시)**  | 낮음 | 높음 |
| **메모리 요구량** | 낮음 | 높음 |
| **적합한 환경** | 오류율이 낮은 채널 | 오류율이 높은 채널 |
