# 3.3 기본 데이터 링크 프로토콜 (Elementary Data Link Protocols)

프로토콜이라는 주제를 소개하기 위해, 복잡도가 점차 증가하는 세 가지 프로토콜을 살펴봅니다.

### 통신 모델의 기본 가정

물리 계층(physical layer), 데이터 링크 계층(data link layer), 네트워크 계층(network layer)은 메시지를 주고받으며 통신하는 **독립적인 프로세스**라고 가정합니다. 일반적인 구현에서 물리 계층 프로세스와 데이터 링크 계층 프로세스의 일부는 **NIC(Network Interface Card)** 라 불리는 전용 하드웨어에서 실행됩니다. 데이터 링크 계층 프로세스의 나머지 부분과 네트워크 계층 프로세스는 메인 CPU에서 운영 체제의 일부로 실행되며, 링크 계층 프로세스의 소프트웨어는 흔히 **디바이스 드라이버(device driver)**  형태를 취합니다.

또 다른 핵심 가정은, 기계 A가 신뢰성 있는 연결 지향 서비스(reliable, connection-oriented service)를 사용하여 기계 B에 긴 데이터 스트림을 보내려 한다는 것입니다. A는 보낼 데이터가 무한히 있고 데이터가 생성될 때까지 기다릴 필요가 없다고 가정합니다. 또한 기계가 충돌하지 않는다고 가정합니다.

데이터 링크 계층이 패킷을 받아들이면 그것을 프레임에 캡슐화하여 데이터 링크 헤더와 트레일러를 추가합니다. 프레임은 내장된 패킷, 일부 제어 정보(헤더 내), 그리고 체크섬(트레일러 내)으로 구성됩니다. **어떤 경우에도 프레임 헤더가 네트워크 계층에 전달되어서는 안 됩니다.** 이는 네트워크와 데이터 링크 프로토콜을 완전히 분리하기 위해서입니다.

### 데이터 구조 및 정의

다섯 가지 데이터 구조가 정의되어 있습니다:

- **`boolean`**: `true`와 `false` 값을 가질 수 있는 열거형 타입입니다.
- **`seq_nr`**: 프레임을 구별하기 위해 번호를 매기는 작은 정수입니다. 순서 번호는 0부터 `MAX_SEQ`까지입니다.
- **`packet`**: 네트워크 계층과 데이터 링크 계층 사이에서 교환되는 정보의 단위입니다.
- **`frame_kind`**: 프레임의 종류(data, ack, nak)를 나타냅니다.
- **`frame`**: 네 개의 필드로 구성됩니다: `kind`, `seq`, `ack`, `info`.
  - **`kind` 필드**: 프레임에 데이터가 있는지를 알려줍니다.
  - **`seq`와 `ack` 필드**: 순서 번호와 확인 응답 번호에 사용됩니다.
  - **`info` 필드**: 데이터 프레임의 경우 단일 패킷을 포함합니다.

### 라이브러리 프로시저 (유틸리티 함수들)

- **`wait_for_event`**: 무언가가 일어나기를 기다립니다.
- **`to_network_layer` / `from_network_layer`**: 네트워크 계층과의 인터페이스를 담당합니다.
- **`from_physical_layer` / `to_physical_layer`**: 물리 계층과의 인터페이스를 담당합니다.
- **`start_timer` / `stop_timer`**: 타이머를 시작/중지합니다.
- **`start_ack_timer` / `stop_ack_timer`**: 보조 타이머를 제어합니다.
- **`enable_network_layer` / `disable_network_layer`**: 네트워크 계층의 이벤트 발생을 허용/금지합니다.
- **매크로 `inc(k)`**: `k`를 순환적으로 1 증가시킵니다 (`MAX_SEQ` 다음은 0).

### 3.3.1 유토피아 단방향 프로토콜 (A Utopian Simplex Protocol)

가능한 한 단순한 프로토콜입니다. 데이터는 **한 방향으로만** 전송됩니다. 통신 채널은 프레임을 절대 손상시키거나 손실하지 않으며, 수신자는 무한한 버퍼 공간과 처리 능력을 가집니다.

#### 프로토콜의 동작 방식

**송신자**는 무한 `while` 루프에서 가능한 한 빠르게 데이터를 전송합니다:
1. 네트워크 계층에서 패킷을 가져옵니다.
2. 프레임을 구성합니다.
3. 프레임을 전송합니다.

**수신자**도 단순합니다:
1. 프레임 도착을 기다립니다.
2. 물리 계층에서 프레임을 가져옵니다.
3. 데이터를 네트워크 계층에 전달합니다.

이 프로토콜에서는 순서 번호, 확인 응답, 흐름 제어, 오류 정정이 전혀 없습니다. 완전히 비현실적이지만 기본 구조를 보여줍니다.

### 3.3.2 오류 없는 채널을 위한 단방향 정지-대기 프로토콜 (A Simplex Stop-and-Wait Protocol for an Error-Free Channel)

수신자가 처리할 수 있는 것보다 더 빠르게 프레임을 보내 수신자가 넘치는(flooding) 문제를 해결합니다. 채널은 여전히 오류가 없다고 가정하며, 데이터는 단방향입니다.

#### 해결 방법: 피드백 기반 접근

수신자가 패킷을 네트워크 계층에 전달한 후, 송신자에게 작은 더미(dummy) 프레임을 보내어 다음 프레임을 전송해도 좋다는 허가를 줍니다. 송신자가 한 프레임을 보내고 나서 확인 응답을 기다리는 프로토콜을 **정지-대기(stop-and-wait)** 라고 부릅니다.

이 예에서 데이터 트래픽은 단방향이지만, 프레임은 **양방향**으로 이동합니다. **반이중(half-duplex)**  물리 채널로도 충분합니다.

### 3.3.3 잡음이 있는 채널을 위한 단방향 정지-대기 프로토콜 (A Simplex Stop-and-Wait Protocol for a Noisy Channel)

오류가 발생하는 통신 채널의 정상적인 상황을 고려합니다. 프레임은 **손상되거나 완전히 손실**될 수 있습니다.

#### 치명적 결함의 시나리오

단순히 타이머를 추가하는 것만으로는 부족합니다. 확인 응답이 손실되면 송신자가 프레임을 재전송하고, 수신자가 **중복 프레임을 새 프레임으로 착각**하여 네트워크 계층에 두 번 전달하는 문제가 발생합니다.

#### 해결책: 순서 번호(Sequence Number)

송신자가 보내는 각 프레임의 헤더에 **순서 번호(sequence number)** 를 넣습니다. 수신자는 도착하는 프레임의 순서 번호를 확인하여 새 프레임인지 중복인지 판단합니다.

#### 최소 순서 번호: 1비트

이 프로토콜에서 유일한 모호성은 프레임 *m*과 그 직접적인 후속 프레임 *m* + 1 사이에 있으므로, **1비트 순서 번호(0 또는 1)** 만으로 충분합니다.

#### ARQ / PAR

이러한 프로토콜을 **ARQ(Automatic Repeat reQuest)**  또는 **PAR(Positive Acknowledgement with Retransmission)** 이라고 합니다.

**송신자의 동작:**
- 프레임 전송 후 타이머를 시작합니다.
- 세 가지 가능성: 유효한 확인 응답 도착(다음 프레임으로 진행), 손상된 확인 응답(재전송), 타이머 만료(재전송)

**수신자의 동작:**
- 유효한 프레임이 도착하면 순서 번호를 확인합니다.
- 기대한 순서 번호면 네트워크 계층에 전달하고, 중복이면 폐기합니다.
- 두 경우 모두 확인 응답을 전송합니다.
