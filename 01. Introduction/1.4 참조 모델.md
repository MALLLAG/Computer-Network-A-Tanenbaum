## 1.4 참조 모델 (Reference Models)

두 가지 중요한 네트워크 아키텍처를 논의하겠습니다: **OSI 참조 모델(OSI reference model)** 과 **TCP/IP 참조 모델(TCP/IP reference model)** 입니다. OSI 모델과 관련된 *프로토콜*들은 더 이상 사용되지 않지만, *모델* 자체는 여전히 유효합니다. TCP/IP 모델은 반대의 특성을 가지고 있습니다: 모델 자체는 별로 쓸모가 없지만 프로토콜들은 널리 사용되고 있습니다.

### 1.4.1 OSI 참조 모델 (The OSI Reference Model)

OSI 모델은 **ISO(International Standards Organization, 국제표준화기구)** 가 개발한 제안에 기반하고 있습니다. **개방형 시스템(open systems)** , 즉 다른 시스템과의 통신을 위해 개방된 시스템을 연결하는 것을 다루기 때문에 ISO **OSI**(**Open Systems Interconnection**) 참조 모델이라고 불립니다.

OSI 모델은 7개의 계층을 가지고 있습니다. 7개의 계층에 도달하기 위해 적용된 원칙들:

1. 서로 다른 수준의 추상화가 필요한 곳에 계층을 만들어야 합니다.
2. 각 계층은 잘 정의된 기능을 수행해야 합니다.
3. 각 계층의 기능은 국제적으로 표준화된 프로토콜을 정의하는 것을 염두에 두고 선택해야 합니다.
4. 계층 경계는 인터페이스를 통과하는 정보 흐름을 최소화하도록 선택해야 합니다.
5. 계층의 수는 충분히 많되, 아키텍처가 다루기 힘들 정도로 많아서는 안 됩니다.

#### 물리 계층 (The Physical Layer)

**물리 계층(physical layer)** 은 통신 채널을 통해 원시 비트(raw bit)를 전송하는 것과 관련됩니다. 1과 0을 나타내기 위해 어떤 전기 신호를 사용해야 하는지, 비트가 몇 나노초 동안 지속되는지, 전송이 양방향으로 동시에 진행될 수 있는지 등을 다룹니다.

#### 데이터 링크 계층 (The Data Link Layer)

**데이터 링크 계층(data link layer)** 의 주요 임무는 원시 전송 설비를 전송 오류가 없는 것처럼 보이는 회선으로 변환하는 것입니다. 송신자가 입력 데이터를 **데이터 프레임(data frames)** 으로 분할하고 프레임을 순차적으로 전송합니다. 수신자는 **확인 프레임(acknowledgement frame)** 을 보내어 확인합니다. 브로드캐스트 네트워크에서는 **매체 접근 제어(medium access control)**  부계층이 공유 채널 접근을 제어합니다.

#### 네트워크 계층 (The Network Layer)

**네트워크 계층(network layer)** 은 서브넷의 운영을 제어합니다. 핵심 설계 이슈는 패킷이 출발지에서 목적지까지 어떻게 라우팅되는지를 결정하는 것입니다. **혼잡(congestion)**  처리도 네트워크 계층의 책임입니다. 이질적인(heterogeneous) 네트워크들이 상호 연결될 수 있도록 하는 것이 네트워크 계층의 역할입니다.

#### 전송 계층 (The Transport Layer)

**전송 계층(transport layer)** 의 기본 기능은 상위 계층으로부터 데이터를 받아들이고, 필요한 경우 더 작은 단위로 분할하여 네트워크 계층에 전달하며, 모든 조각이 올바르게 도착하도록 보장하는 것입니다. 전송 계층은 진정한 **종단 간(end-to-end)**  계층입니다.

#### 세션 계층 (The Session Layer)

**세션 계층(session layer)** 은 **대화 제어(dialog control)** , **토큰 관리(token management)** , **동기화(synchronization)**  등의 서비스를 제공합니다.

#### 표현 계층 (The Presentation Layer)

**표현 계층(presentation layer)** 은 전송되는 정보의 구문(syntax)과 의미론(semantics)에 관여합니다. 서로 다른 내부 데이터 표현을 사용하는 컴퓨터들이 통신할 수 있도록 추상 데이터 구조를 정의하고 표준 인코딩을 관리합니다.

#### 응용 계층 (The Application Layer)

**응용 계층(application layer)** 은 사용자가 필요로 하는 다양한 프로토콜을 포함합니다. **HTTP**(**HyperText Transfer Protocol**)는 월드 와이드 웹의 기초입니다. 다른 응용 프로토콜은 파일 전송, 전자 메일, 네트워크 뉴스 등에 사용됩니다.

### 1.4.2 TCP/IP 참조 모델 (The TCP/IP Reference Model)

ARPANET과 그 후계자인 인터넷에서 사용되는 참조 모델입니다. **DoD(U.S. Department of Defense, 미국 국방부)** 가 후원하는 연구 네트워크였으며, 처음부터 여러 네트워크를 매끄럽게 연결하는 능력이 주요 설계 목표였습니다.

#### 링크 계층 (The Link Layer)

**링크 계층(link layer)** 은 호스트와 전송 링크 사이의 인터페이스에 가깝습니다.

#### 인터넷 계층 (The Internet Layer)

**인터넷 계층(internet layer)** 은 전체 아키텍처를 하나로 묶는 핵심 축입니다. 호스트가 어떤 네트워크에든 패킷을 주입하고 그 패킷들이 목적지까지 독립적으로 이동하도록 허용합니다. 인터넷 계층은 **IP**(**Internet Protocol**)와 동반 프로토콜인 **ICMP**(**Internet Control Message Protocol**)를 정의합니다.

#### 전송 계층 (The Transport Layer)

두 가지 종단 간 전송 프로토콜이 정의되어 있습니다:
- **TCP**(**Transmission Control Protocol**): 신뢰성 있는 연결 지향 프로토콜로, 바이트 스트림을 오류 없이 전달합니다. **흐름 제어(flow control)** 도 처리합니다.
- **UDP**(**User Datagram Protocol**): 비신뢰성, 비연결형 프로토콜로, 일회성 요청-응답 질의와 실시간 응용에 사용됩니다.

<img width="670" height="333" alt="Image" src="https://github.com/user-attachments/assets/3af39b80-608e-4947-99f7-d5c4bd1a1c42" />

#### 응용 계층 (The Application Layer)

TCP/IP 모델에는 세션 계층이나 표현 계층이 없습니다. 초기 프로토콜에는 TELNET, FTP, SMTP가 포함되었으며, 이후 **DNS**, **HTTP**, **RTP** 등이 추가되었습니다.

### 1.4.3 이 책에서 사용하는 모델 (The Model Used in This Book)

이 책의 프레임워크로 5개 계층의 하이브리드(혼합) 모델을 사용합니다: **물리(Physical)** , **링크(Link)** , **네트워크(Network)** , **전송(Transport)** , **응용(Application)**  계층입니다.

### 1.4.4 OSI와 TCP/IP 참조 모델의 비교 (A Comparison of the OSI and TCP/IP Reference Models)

두 모델의 공통점: 독립적인 프로토콜의 스택 개념에 기반하며, 계층의 기능도 대체로 유사합니다.

OSI 모델의 핵심 세 가지 개념:
1. **서비스(Services)** : 계층이 무엇을 하는지를 정의
2. **인터페이스(Interfaces)** : 상위의 프로세스에게 해당 계층에 어떻게 접근하는지를 정의
3. **프로토콜(Protocols)** : 제공되는 서비스를 달성하기 위해 자유롭게 선택 가능

주요 차이점: OSI 모델은 7개 계층, TCP/IP 모델은 4개 계층입니다. OSI 모델은 네트워크 계층에서 비연결형과 연결 지향형 통신을 모두 지원하지만, TCP/IP 모델은 네트워크 계층에서 비연결형만을 지원하고 전송 계층에서 둘 다 지원합니다.

### 1.4.5 OSI 모델과 프로토콜에 대한 비판 (A Critique of the OSI Model and Protocols)

OSI가 실패한 이유:

1. **나쁜 타이밍**: 경쟁하는 TCP/IP 프로토콜이 이미 연구 대학들에서 광범위하게 사용되고 있었습니다.
2. **나쁜 기술**: 7개 계층 중 세션과 표현 계층은 거의 비어 있고, 데이터 링크와 네트워크 계층은 과도하게 채워져 있었습니다. 모델은 극도로 복잡했습니다.
3. **나쁜 구현**: 초기 구현이 거대하고, 다루기 힘들고, 느렸습니다.
4. **나쁜 정치**: OSI는 정부 관료의 산물로 인식되어 부정적 이미지가 있었습니다.

### 1.4.6 TCP/IP 참조 모델에 대한 비판 (A Critique of the TCP/IP Reference Model)

TCP/IP 모델의 문제점:

1. 서비스, 인터페이스, 프로토콜의 개념을 명확하게 구분하지 않습니다.
2. 전혀 일반적이지 않으며 TCP/IP 이외의 다른 프로토콜 스택을 기술하는 데 적합하지 않습니다.
3. 링크 계층은 일반적인 의미의 계층이 아니라 인터페이스에 가깝습니다.
4. 물리 계층과 데이터 링크 계층을 구분하지 않습니다.
5. 일부 프로토콜은 임시방편(ad hoc)으로 만들어졌습니다.
