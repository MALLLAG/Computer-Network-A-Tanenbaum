# 전송계층 (Transport Layer) 정리

---

## 1. Circuit Switching vs Packet Switching

### Circuit Switching (회선 교환)

- 통신 전에 **전용 경로(회선)를 미리 설정**한 후 데이터를 전송하는 방식
- 전화망(PSTN)이 대표적인 예시

**동작 과정:**
1. **연결 설정 (Setup):** 송신자 → 수신자까지의 전용 경로를 확보
2. **데이터 전송:** 확보된 경로를 통해 데이터 전송
3. **연결 해제 (Teardown):** 통신 종료 후 회선 반환

**장점:**
- 전용 경로이므로 **일정한 전송 속도(QoS)** 보장
- 전송 지연이 예측 가능하고 안정적

**단점:**
- 회선 설정 시간이 필요 (초기 지연)
- 데이터를 보내지 않는 동안에도 **회선을 독점** → 자원 낭비
- 사용자가 많아지면 회선 부족 문제 발생

**다중화 방식:**
- **FDM (주파수 분할 다중화):** 주파수 대역을 나누어 할당
- **TDM (시간 분할 다중화):** 시간 슬롯을 나누어 할당

---

### Packet Switching (패킷 교환)

- 데이터를 **패킷 단위로 분할**하여 독립적으로 전송하는 방식
- 인터넷이 대표적인 예시

**동작 과정:**
1. 데이터를 패킷으로 분할
2. 각 패킷이 **독립적으로 라우팅**되어 목적지로 전송
3. 수신 측에서 패킷을 재조립

**장점:**
- 회선을 공유하므로 **자원 활용 효율이 높음**
- 더 많은 사용자 수용 가능
- 전용 회선 설정 불필요

**단점:**
- **큐잉 지연 (Queuing Delay)** 발생 가능
- 패킷 손실 가능 (버퍼 오버플로우)
- 전송 순서 보장 안 됨 (별도 처리 필요)

### 비교 요약

| 항목 | Circuit Switching | Packet Switching |
|------|------------------|-----------------|
| 경로 설정 | 사전에 전용 경로 확보 | 패킷마다 독립 라우팅 |
| 자원 사용 | 독점 (비효율적) | 공유 (효율적) |
| 지연 | 일정하고 예측 가능 | 가변적 (큐잉 지연) |
| 신뢰성 | 높음 | 패킷 손실 가능 |
| 적합한 용도 | 실시간 음성 통화 | 인터넷 데이터 통신 |

## 2. TCP Timer

TCP는 신뢰성 있는 전송을 위해 여러 종류의 타이머를 사용한다.

### 2.1 Retransmission Timer (재전송 타이머)

- 세그먼트를 전송한 후 ACK를 기다리는 타이머
- **타임아웃(RTO)** 내에 ACK가 도착하지 않으면 **재전송**

**RTO 계산 방식:**

```
SampleRTT: 실제 측정한 왕복 시간
EstimatedRTT = (1 - α) × EstimatedRTT + α × SampleRTT   (α = 0.125)
DevRTT = (1 - β) × DevRTT + β × |SampleRTT - EstimatedRTT|   (β = 0.25)
TimeoutInterval (RTO) = EstimatedRTT + 4 × DevRTT
```

**핵심 포인트:**
- RTT의 변동성(DevRTT)을 반영하여 동적으로 조절
- **EWMA (Exponentially Weighted Moving Average)** 방식 사용
- 타임아웃 발생 시 RTO를 **2배로 증가** (Exponential Backoff)

### 2.2 Persistence Timer (지속 타이머)

- 수신 측의 **윈도우 크기가 0**일 때 사용
- 송신 측이 주기적으로 **Probe 세그먼트**를 전송하여 윈도우가 열렸는지 확인
- **교착 상태(Deadlock) 방지** 목적

### 2.3 Keep-Alive Timer (연결 유지 타이머)

- 일정 시간 동안 데이터 교환이 없을 때 **연결이 살아있는지 확인**
- 보통 **2시간** 동안 데이터가 없으면 Probe 전송

### 2.4 TIME_WAIT Timer (2MSL 타이머)

- TCP 연결 종료 시 **마지막 ACK 전송 후 대기**하는 타이머
- **2 × MSL (Maximum Segment Lifetime)** 동안 대기
- 목적:
  - 마지막 ACK가 손실될 경우 재전송 대응
  - 이전 연결의 지연 패킷이 새 연결에 영향을 주는 것 방지

## 3. Go-Back-N (GBN) vs Selective Repeat (SR)

두 프로토콜 모두 **슬라이딩 윈도우** 기반의 파이프라인 프로토콜이다.

### 3.1 Go-Back-N (GBN)

**핵심 원리:** 오류 발생 시 해당 패킷부터 **윈도우 내 모든 패킷을 재전송**

**송신 측:**
- 윈도우 크기: N (최대 N개의 미확인 패킷 허용)
- 타이머: **가장 오래된 미확인 패킷에 대해 하나의 타이머**만 사용
- 타임아웃 시: 윈도우 내 **모든 미확인 패킷 재전송**

**수신 측:**
- **순서대로(in-order)만 수락**
- 순서에 맞지 않는 패킷은 **버림 (버퍼링 안 함)**
- **Cumulative ACK** 사용: ACK(n) = "n번까지 모두 잘 받았다"

**예시:**
```
  윈도우 크기 N = 4

  송신측:  [1][2][3][4] | [5][6][7]...
            ^^^^^^^^^
            윈도우 범위

  패킷 2 손실 시:
  송신: 1  2  3  4
          ✗
  수신: ACK1  (3,4는 버림 → 누적 ACK1 반복)

  → 타임아웃 후 2, 3, 4 모두 재전송
```

**윈도우 크기 제한:** N ≤ 2^k - 1 (k = 시퀀스 번호 비트 수)

---

### 3.2 Selective Repeat (SR)

**핵심 원리:** 오류가 발생한 **해당 패킷만 선택적으로 재전송**

**송신 측:**
- 윈도우 크기: N
- **각 패킷마다 개별 타이머** 사용
- 타임아웃 시: **해당 패킷만 재전송**

**수신 측:**
- 순서에 맞지 않는 패킷도 **버퍼에 저장**
- **Individual ACK** 사용: ACK(n) = "n번 패킷만 잘 받았다"
- 빠진 패킷이 도착하면 버퍼의 패킷과 함께 상위 계층에 전달

**예시:**
```
  윈도우 크기 N = 4

  패킷 2 손실 시:
  송신: 1  2  3  4
          ✗
  수신: ACK1  ACK3  ACK4  (3,4는 버퍼에 저장)

  → 타임아웃 후 패킷 2만 재전송
  → 패킷 2 도착 → 버퍼의 3,4와 합쳐서 상위 계층 전달
```

**윈도우 크기 제한:** N ≤ 2^(k-1) (k = 시퀀스 번호 비트 수)

---

### 비교 요약

| 항목 | Go-Back-N | Selective Repeat |
|------|-----------|-----------------|
| 재전송 범위 | 윈도우 내 전체 재전송 | 손실된 패킷만 재전송 |
| 수신 측 버퍼 | 불필요 (순서대로만 수락) | 필요 (비순서 패킷 버퍼링) |
| ACK 방식 | Cumulative ACK | Individual ACK |
| 타이머 | 1개 (가장 오래된 패킷) | 패킷마다 개별 타이머 |
| 대역폭 효율 | 낮음 (불필요한 재전송 많음) | 높음 (필요한 것만 재전송) |
| 구현 복잡도 | 단순 | 복잡 |
| 윈도우 크기 | ≤ 2^k - 1 | ≤ 2^(k-1) |

## 4. Fast Retransmit (빠른 재전송)

### 문제 상황

- 일반적인 재전송은 **타임아웃을 기다려야** 함
- RTO가 길면 재전송까지의 지연이 큼 → 성능 저하

### Fast Retransmit 메커니즘

- **3개의 중복 ACK (Triple Duplicate ACK)** 를 수신하면 타임아웃을 기다리지 않고 **즉시 재전송**
- 총 4개의 동일한 ACK (원본 1개 + 중복 3개)를 받으면 트리거

**동작 과정:**
```
송신: Seq=100, Seq=200, Seq=300, Seq=400, Seq=500
                  ↑ Seq=200 손실!

수신:
  Seq=100 수신 → ACK 200
  Seq=200 손실
  Seq=300 수신 → ACK 200  (중복 ACK 1)
  Seq=400 수신 → ACK 200  (중복 ACK 2)
  Seq=500 수신 → ACK 200  (중복 ACK 3) ← 3 Duplicate ACK!

송신: 타임아웃 전에 Seq=200 즉시 재전송!
```

### 왜 3개인가?

- 1~2개의 중복 ACK는 **패킷 재정렬(reordering)** 로 발생할 수 있음
- 3개의 중복 ACK는 **패킷 손실일 확률이 매우 높음**
- 불필요한 재전송과 빠른 대응 사이의 **적절한 균형점**

---

## 5. Triple Duplicate ACK (3중 중복 ACK)

### 개념

```
송신자                          수신자  
|--- Seq 100 (100 bytes) --->  |  
|--- Seq 200 (손실!) ------X    |  
|--- Seq 300 ------------->    |  → ACK 200 (dup #1) "200번 달라"  
|--- Seq 400 ------------->    |  → ACK 200 (dup #2) "200번 달라"  
|--- Seq 500 ------------->    |  → ACK 200 (dup #3) "200번 달라"  
|                              |  
| ← 3 duplicate ACKs 감지!      |  
|                              |  
|--- Seq 200 (재전송!) ---->     |  → ACK 600 (누적 ACK)  
```

- 수신 측이 **기대하는 시퀀스 번호가 아닌 패킷**을 받으면 **가장 마지막으로 정상 수신한 시퀀스 번호에 대한 ACK를 재전송**
- 동일한 ACK가 **원본 포함 총 4번** 송신 측에 도달

### 발생 원인

1. **패킷 손실:** 중간 패킷이 네트워크에서 유실
2. **패킷 재정렬:** 뒤의 패킷이 먼저 도착 (이 경우 보통 1~2개의 중복 ACK만 발생)

### 의미와 활용

- **Fast Retransmit 트리거:** 손실된 세그먼트 즉시 재전송
- **혼잡 감지 신호:** 네트워크에 혼잡이 발생했을 가능성
  - 하지만 타임아웃보다는 **덜 심각한 혼잡**으로 판단
  - 패킷이 아예 전달이 안 된 것이 아니라, 일부는 도착했으므로

## 6. TCP Tahoe vs TCP Reno

두 방식 모두 TCP의 **혼잡 제어 알고리즘**이다.

### 공통 개념

- **cwnd (Congestion Window):** 송신 측이 관리하는 혼잡 윈도우 크기
- **ssthresh (Slow Start Threshold):** Slow Start에서 Congestion Avoidance로 전환하는 임계값
- **MSS (Maximum Segment Size):** 최대 세그먼트 크기

### 공통 단계

#### Slow Start (느린 시작)
- cwnd = 1 MSS에서 시작
- ACK를 받을 때마다 cwnd를 **1 MSS 증가** → 매 RTT마다 cwnd **2배** (지수 증가)
- cwnd가 ssthresh에 도달하면 → Congestion Avoidance로 전환

#### Congestion Avoidance (혼잡 회피)
- 매 RTT마다 cwnd를 **1 MSS 증가** (선형 증가, AIMD의 AI 부분)
- 혼잡이 감지될 때까지 계속

---

### 6.1 TCP Tahoe

**혼잡 감지 시 동작 (타임아웃 OR 3 Duplicate ACK):**
1. ssthresh = cwnd / 2
2. **cwnd = 1 MSS** (처음부터 다시 시작)
3. **Slow Start부터 재시작**

```
cwnd
 ↑
 |        /\
 |       /  \
 |      /    \
 |     /      \         /\
 |    /        \       /  \
 |   /          \     /    \
 |  /            \   /      \
 | /              \ /        \
 |/                V          \
 +------------------------→ time
   Slow    CA   손실!  Slow  CA
   Start        cwnd=1  Start
```

**특징:**
- 구현이 단순
- 손실 시 항상 cwnd를 1로 리셋 → **회복이 느림**
- 타임아웃과 3 Duplicate ACK를 **동일하게 처리**

---

### 6.2 TCP Reno

**혼잡 감지 시 동작 — 두 가지 경우를 구분:**

#### Case 1: 타임아웃 발생 (심각한 혼잡)
- Tahoe와 동일하게 처리
1. ssthresh = cwnd / 2
2. cwnd = 1 MSS
3. Slow Start부터 재시작

#### Case 2: 3 Duplicate ACK (경미한 혼잡) → **Fast Recovery**
1. ssthresh = cwnd / 2
2. **cwnd = ssthresh + 3 MSS** (절반으로 줄이되, 1로 리셋하지 않음)
3. **Fast Recovery** 상태 진입
4. 중복 ACK가 추가로 올 때마다 cwnd += 1 MSS
5. 새로운 ACK(손실 패킷에 대한 ACK) 도착 시 → cwnd = ssthresh로 설정 → **Congestion Avoidance로 전환**

```
cwnd
 ↑
 |        /\
 |       /  \
 |      /    \
 |     /      +----         ← Fast Recovery (cwnd 절반)
 |    /        \    \
 |   /          \    ----    ← Congestion Avoidance
 |  /            \
 | /              (타임아웃 시에만 cwnd=1)
 |/
 +------------------------→ time
```

**특징:**
- 3 Duplicate ACK 시 cwnd를 1로 리셋하지 않음 → **빠른 회복**
- 타임아웃과 3 Duplicate ACK를 **다르게 처리**
- Tahoe보다 **높은 처리량(throughput)** 달성

---

### Tahoe vs Reno 비교

| 항목 | TCP Tahoe | TCP Reno |
|------|-----------|----------|
| 3 Dup ACK 시 cwnd | **1 MSS** (리셋) | **cwnd/2 + 3** (절반) |
| 타임아웃 시 cwnd | 1 MSS | 1 MSS |
| Fast Recovery | **없음** | **있음** |
| 3 Dup ACK 후 상태 | Slow Start | Fast Recovery → CA |
| 타임아웃 후 상태 | Slow Start | Slow Start |
| 회복 속도 | 느림 | 빠름 |
| 처리량 | 상대적으로 낮음 | 상대적으로 높음 |

## 7. Flow Control vs Congestion Control

### 7.1 Flow Control (흐름 제어)

**목적:** **수신 측의 버퍼 오버플로우 방지**

- **송신 측이 수신 측의 처리 속도에 맞춰** 전송 속도를 조절
- **End-to-End** 문제 (송신자 ↔ 수신자 사이의 문제)

**메커니즘: Sliding Window (rwnd)**
- 수신 측이 TCP 헤더의 **Window Size 필드**를 통해 **rwnd (Receive Window)** 를 송신 측에 알려줌
- `rwnd = RcvBuffer - (LastByteRcvd - LastByteRead)`
- 송신 측은 `LastByteSent - LastByteAcked ≤ rwnd` 를 유지

**동작 예시:**
```
수신 측 버퍼 크기: 4000 bytes
현재 버퍼 사용량: 3000 bytes
→ rwnd = 1000 bytes (수신 가능한 양)
→ 송신 측은 최대 1000 bytes까지만 전송 가능

버퍼가 가득 차면 → rwnd = 0 → 송신 중단
→ Persistence Timer로 주기적 Probe 전송
```

---

### 7.2 Congestion Control (혼잡 제어)

**목적:** **네트워크의 과부하 방지**

- **네트워크 상태를 고려하여** 전송 속도를 조절
- **네트워크 전체**의 문제 (여러 송신자가 네트워크를 공유)

**메커니즘: Congestion Window (cwnd)**
- 송신 측이 네트워크 혼잡도를 추측하여 **cwnd** 를 조절
- 혼잡 감지 방법: **패킷 손실** (타임아웃 또는 3 Duplicate ACK)

**알고리즘:**
1. **Slow Start:** cwnd 지수적 증가
2. **Congestion Avoidance:** cwnd 선형적 증가 (AIMD)
3. **Fast Recovery:** (Reno) 3 Dup ACK 시 빠른 회복

---

### 실제 전송 가능량

TCP에서 실제로 전송 가능한 양은 두 제어의 **최솟값**으로 결정된다:

```
실제 전송 가능량 = min(cwnd, rwnd)
```

---

### 비교 요약

| 항목 | Flow Control | Congestion Control |
|------|-------------|-------------------|
| 목적 | 수신 측 보호 | 네트워크 보호 |
| 제어 대상 | 수신 측 버퍼 용량 | 네트워크 혼잡도 |
| 문제 범위 | End-to-End (양 끝단) | Network-wide (네트워크 전체) |
| 윈도우 | rwnd (Receive Window) | cwnd (Congestion Window) |
