# 8.9 웹 보안 (Web Security)

지금까지 보안이 필요한 두 가지 중요한 영역인 통신과 이메일을 학습하였습니다. 이것들을 수프와 전채 요리에 비유할 수 있습니다. 이제 메인 요리인 **웹 보안(Web security)** 을 다룰 차례입니다. 웹은 요즘 대부분의 Trudy들이 모여 있고 그들의 더러운 작업을 수행하는 곳입니다. 다음 절에서는 웹 보안과 관련된 몇 가지 문제와 이슈를 살펴보겠습니다.

웹 보안은 크게 **세 부분**으로 나눌 수 있습니다:

1. **첫째**, 객체의 이름을 어떻게 안전하게 지정하는가?
2. **둘째**, 어떻게 안전하고 인증된 연결을 확립하는가?
3. **셋째**, 웹 사이트가 클라이언트에게 실행 가능한 코드 조각을 보낼 때 어떤 일이 발생하는가?

몇 가지 위협을 살펴본 후, 이 모든 이슈들을 검토하겠습니다.

---

## 8.9.1 위협 (Threats)

웹 사이트 보안 문제에 대한 뉴스는 거의 매주 신문에 보도됩니다. 상황은 정말로 매우 심각합니다. 이미 발생한 사례 몇 가지를 살펴보겠습니다.

### 웹 사이트 변조 (Defacement)

수많은 조직의 홈 페이지가 공격당하고 크래커들이 원하는 새로운 홈 페이지로 교체되었습니다. (대중 언론에서는 컴퓨터에 침입하는 사람들을 "해커(hackers)"라고 부르지만, 많은 프로그래머들은 그 용어를 뛰어난 프로그래머에게 사용합니다. 이 사람들을 "**크래커(crackers)**"라고 부르는 것이 더 적절합니다.) 크래킹된 사이트에는 Yahoo!, 미 육군, CIA, NASA, 그리고 *New York Times*에 속한 사이트들이 포함됩니다. 대부분의 경우, 크래커들은 재미있는 텍스트를 올리고 사이트는 몇 시간 내에 복구되었습니다.

### 서비스 거부 공격 (DDoS)

더 심각한 사례도 있습니다. 많은 사이트들이 **서비스 거부 공격(denial-of-service attacks)** 에 의해 무력화되었는데, 이 공격에서 크래커는 사이트에 트래픽을 폭주시켜 정상적인 요청에 응답할 수 없게 만듭니다. 종종 이 공격은 크래커가 이미 침입한 대량의 머신들로부터 수행되는데, 이를 **DDoS(분산 서비스 거부) 공격**이라고 합니다. 이러한 공격은 너무 흔해져서 더 이상 뉴스에 나오지도 않지만, 공격받은 사이트들에게 수천 달러의 비즈니스 손실을 초래할 수 있습니다.

### 미러 사이트 공격

1999년에 스웨덴의 한 크래커가 Microsoft의 Hotmail 웹 사이트에 침입하여 누구나 Hotmail 사용자의 이름을 입력하면 그 사람의 현재 및 보관된 이메일을 모두 읽을 수 있는 미러 사이트를 만들었습니다.

### 신용카드 도용

또 다른 사례로, 19세의 러시아 크래커 Maxim이 한 전자상거래 웹 사이트에 침입하여 **300,000개의 신용카드 번호**를 도용하였습니다. 그런 다음 사이트 소유자에게 접근하여 **$100,000**을 지불하지 않으면 모든 신용카드 번호를 인터넷에 공개하겠다고 협박하였습니다. 사이트 소유자들은 협박에 굴하지 않았고, 크래커는 실제로 신용카드 번호를 공개하여 많은 무고한 피해자들에게 큰 피해를 입혔습니다.

### 허위 보도 공격

다른 종류의 사례로, 23세의 캘리포니아 대학생이 뉴스 통신사에 허위 보도 자료를 이메일로 보내어, Emulex Corporation이 대규모 분기 손실을 공시할 예정이며 CEO가 즉시 사임한다고 거짓으로 발표하였습니다. 몇 시간 내에 회사의 주가가 **60%** 하락하여 주주들에게 **20억 달러 이상**의 손실을 초래하였습니다. 범인은 발표 직전에 공매도(selling the stock short)를 하여 25만 달러의 이익을 취하였습니다. 이 사건이 웹 사이트 침입은 아니었지만, 어떤 대기업의 홈 페이지에 이런 공지를 올리면 유사한 효과가 발생할 것이 분명합니다.

이와 같은 사례는 더 많이 열거할 수 있지만, 이제 웹 보안과 관련된 기술적 이슈를 검토할 시간입니다. 보안 문제에 대한 추가 정보는 Anderson (2008a), Stuttard and Pinto (2007), Schneier (2004)를 참조하시기 바랍니다.

## 8.9.2 안전한 이름 지정 (Secure Naming)

매우 기본적인 것에서 시작하겠습니다: Alice가 Bob의 웹 사이트를 방문하려고 합니다. 그녀가 Bob의 URL을 브라우저에 입력하고 몇 초 후에 웹 페이지가 나타납니다. 하지만 그것이 정말 Bob의 것인가요? 아마도 그렇고 아마도 아닐 수 있습니다. Trudy가 또 다시 속임수를 쓰고 있을 수 있습니다.

### 중간자 공격 (Man-in-the-Middle Attack)

예를 들어, Trudy가 Alice의 모든 발신 패킷을 가로채 검토하고 있을 수 있습니다. Bob의 웹 사이트로 향하는 HTTP *GET* 요청을 캡처하면, 직접 Bob의 웹 사이트에 가서 페이지를 가져오고, 원하는 대로 수정하여 가짜 페이지를 Alice에게 반환할 수 있습니다. Alice는 전혀 눈치채지 못할 것입니다. 더 나쁜 것은, Trudy가 Bob의 전자 상점의 가격을 대폭 할인하여 상품을 매우 매력적으로 보이게 만들 수 있고, 이를 통해 Alice가 "Bob"에게 물건을 사려고 신용카드 번호를 보내도록 속일 수 있습니다.

이 고전적인 중간자 공격(man-in-the-middle attack)의 한 가지 단점은 Trudy가 Alice의 발신 트래픽을 가로채고 수신 트래픽을 위조할 수 있는 위치에 있어야 한다는 것입니다. 실제로 Alice의 전화선이나 Bob의 전화선을 도청해야 하는데, 광섬유 백본의 도청은 상당히 어렵습니다. 능동적인 도청은 확실히 가능하지만, 상당한 작업량이 필요하고, Trudy가 영리하기는 하지만 게으르기도 합니다. 게다가, Alice를 속이는 더 쉬운 방법이 있습니다.

---

### DNS 스푸핑 (DNS Spoofing)

Trudy의 한 가지 방법은 **DNS 시스템** 또는 Alice의 ISP에 있는 DNS 캐시를 크래킹하여 Bob의 IP 주소(예: 36.1.2.3)를 자신(Trudy)의 IP 주소(예: 42.9.9.9)로 교체하는 것입니다.

#### 정상적인 상황 vs. 공격 상황

<img width="748" height="459" alt="Image" src="https://github.com/user-attachments/assets/e4e59e65-18c8-4127-8709-7af5b8ddb19a" />

> **(a) 정상 상황**:
> 1. Alice가 DNS 서버에 "Bob의 IP 주소를 알려주세요"라고 요청합니다.
> 2. DNS 서버가 36.1.2.3 (Bob의 IP 주소)를 응답합니다.
> 3. Alice가 Bob의 웹 서버(36.1.2.3)에 GET index.html을 요청합니다.
> 4. Bob의 홈 페이지가 반환됩니다.
>
> **(b) DNS 서버가 크래킹된 경우**:
> 1. Alice가 크래킹된 DNS 서버에 "Bob의 IP 주소를 알려주세요"라고 요청합니다.
> 2. 크래킹된 DNS 서버가 42.9.9.9 (Trudy의 IP 주소)를 응답합니다.
> 3. Alice가 Trudy의 웹 서버(42.9.9.9)에 GET index.html을 요청합니다.
> 4. Trudy가 만든 가짜 Bob의 홈 페이지가 반환됩니다.

정상적으로 작동해야 하는 방식은 그림 8-46(a)에 설명되어 있습니다. Alice가 (1) Bob의 IP 주소를 DNS에 요청하고, (2) 이를 받고, (3) Bob에게 홈 페이지를 요청하고, (4) 이를 받습니다. Trudy가 Bob의 DNS 레코드를 수정하여 자신의 IP 주소를 대신 넣으면, 그림 8-46(b)의 상황이 됩니다. Alice가 Bob의 IP 주소를 조회하면 Trudy의 것을 받게 되어, Bob을 향한 모든 트래픽이 Trudy에게 갑니다. Trudy는 전화선을 도청하는 수고 없이 **중간자 공격(man-in-the-middle attack)** 을 수행할 수 있습니다. 대신 DNS 서버에 침입하여 하나의 레코드를 변경하면 되는데, 이는 훨씬 더 쉬운 일입니다.

#### DNS 스푸핑의 구체적 방법

Trudy가 어떻게 DNS를 속일 수 있을까요? 간략히 요약하면, Trudy는 Alice의 ISP에 있는 DNS 서버를 속여 Bob의 주소를 조회하는 쿼리를 보내도록 만들 수 있습니다. 불행히도 DNS는 **UDP**를 사용하므로, DNS 서버는 누가 응답을 보냈는지 확인할 실질적인 방법이 없습니다. Trudy는 예상되는 응답을 위조하여 DNS 서버의 캐시에 거짓 IP 주소를 주입함으로써 이 특성을 악용할 수 있습니다.

단순화를 위해, Alice의 ISP가 처음에 Bob의 웹 사이트 *bob.com*에 대한 항목을 가지고 있지 않다고 가정합니다. 있다면, Trudy는 타임아웃될 때까지 기다리거나 다른 트릭을 사용할 수 있습니다.

Trudy는 Alice의 ISP에 *bob.com*의 IP 주소에 대한 조회 요청을 보내어 공격을 시작합니다. 이 DNS 이름에 대한 항목이 없으므로, 캐시 서버는 *com* 도메인의 최상위 서버에 쿼리를 보내게 됩니다. 그러나 Trudy가 *com* 서버보다 먼저 "bob.com은 42.9.9.9이다"라는 거짓 응답을 보내면, 이 거짓 응답이 Alice의 ISP에 먼저 도착하면 캐시에 저장되고, 더 이상 미해결 쿼리가 없으므로 실제 응답은 거부됩니다.

DNS 서버에 거짓 IP 주소를 설치하도록 속이는 것을 **DNS 스푸핑(DNS spoofing)** 이라고 합니다. 의도적으로 거짓 IP 주소를 가진 캐시를 **오염된 캐시(poisoned cache)** 라고 합니다.

#### ISP의 보안 검사 우회

실제로는 그렇게 간단하지 않습니다. **첫째**, Alice의 ISP는 응답이 최상위 서버의 올바른 IP 소스 주소를 가지고 있는지 확인합니다. 하지만 Trudy가 해당 IP 필드에 원하는 것을 넣을 수 있으므로, 최상위 서버의 IP 주소는 공개되어 있기 때문에 이 검사를 쉽게 통과할 수 있습니다.

**둘째**, 어떤 응답이 어떤 요청에 해당하는지 구별하기 위해, 모든 DNS 요청은 **시퀀스 번호(sequence number)** 를 포함합니다. Alice의 ISP를 스푸핑하려면, Trudy는 현재 시퀀스 번호를 알아야 합니다.

현재 시퀀스 번호를 알아내는 가장 쉬운 방법은 Trudy가 직접 도메인을 등록하는 것입니다. 예를 들어 *trudy-the-intruder.com*이라고 합시다. IP 주소도 42.9.9.9라고 가정합니다. 그녀는 또한 새로 만든 도메인을 위한 DNS 서버 *dns.trudy-the-intruder.com*을 생성하며, 이것도 Trudy의 42.9.9.9 IP 주소를 사용합니다(Trudy는 컴퓨터가 하나뿐이기 때문). 이제 Alice의 ISP에 자신의 DNS 서버를 인식시켜야 합니다. *foobar.trudy-the-intruder.com*을 조회하도록 Alice의 ISP에 요청하면 됩니다. 이렇게 하면 Alice의 ISP가 최상위 *com* 서버에 Trudy의 새 도메인을 누가 서비스하는지 물어보게 됩니다.

<img width="733" height="350" alt="Image" src="https://github.com/user-attachments/assets/519b6050-97af-41f3-b31b-15b13929ee7f" />

> 1. *foobar.trudy-the-intruder.com*을 조회합니다 (ISP의 캐시에 강제로 넣기 위해).
> 2. *www.trudy-the-intruder.com*을 조회합니다 (ISP의 다음 시퀀스 번호를 얻기 위해).
> 3. *www.trudy-the-intruder.com*에 대한 요청 (ISP의 다음 시퀀스 번호 n을 포함).
> 4. 토끼처럼 빠르게 *bob.com*을 조회합니다 (5단계에서 ISP가 *com* 서버에 쿼리하도록 강제하기 위해).
> 5. *bob.com*에 대한 정상적인 쿼리, seq = n+1.
> 6. Trudy의 위조 응답: "Bob은 42.9.9.9이다", seq = n+1.
> 7. 실제 응답 (거부됨, 너무 늦음).

*dns.trudy-the-intruder.com*이 Alice의 ISP의 캐시에 안전하게 저장되면, 실제 공격이 시작될 수 있습니다. Trudy는 이제 Alice의 ISP에 *www.trudy-the-intruder.com*을 조회합니다. ISP는 자연스럽게 Trudy의 DNS 서버에 쿼리를 보냅니다. 이 쿼리에는 Trudy가 찾고 있던 **시퀀스 번호**가 포함되어 있습니다. 토끼처럼 빠르게, Trudy는 Alice의 ISP에 Bob을 조회하도록 요청합니다. 그녀는 즉시 자신의 질문에 대한 응답을 보내는데, 마치 최상위 *com* 서버에서 온 것처럼 "bob.com은 42.9.9.9이다"라는 위조 응답을 보냅니다. 이 위조 응답은 방금 받은 시퀀스 번호보다 하나 높은 시퀀스 번호를 포함합니다. 동시에 시퀀스 번호가 두 개 높은 두 번째 위조, 그리고 시퀀스 번호가 점점 증가하는 열두 개 정도를 더 보낼 수 있습니다. 이 중 하나는 반드시 일치하게 됩니다. 나머지는 버려질 것입니다. Trudy의 위조 응답이 도착하면 캐시에 저장됩니다; 실제 응답이 나중에 도착하면, 미해결 쿼리가 없으므로 거부됩니다.

이제 Alice가 *bob.com*을 조회하면, 42.9.9.9(Trudy의 주소)를 사용하라고 안내받습니다. Trudy는 자신의 거실 안락의자에서 성공적인 중간자 공격을 수행한 것입니다.

이 특정 공격은 DNS 서버가 쿼리에 단순히 카운팅하는 것이 아니라 **랜덤 ID**를 사용하도록 함으로써 방어할 수 있지만, 하나의 허점을 막으면 다른 것이 나타나는 것 같습니다. 특히 ID는 16비트에 불과하므로, 컴퓨터가 추측하기에는 매우 쉽습니다.

### 안전한 DNS (Secure DNS / DNSsec)

진짜 문제는 DNS가 인터넷이 몇 백 개의 대학을 위한 연구 시설이었고, Alice도, Bob도, Trudy도 파티에 초대받지 않았던 시기에 설계되었다는 것입니다. 당시에는 보안이 문제가 되지 않았습니다; 인터넷을 작동시키는 것이 관건이었습니다. 환경은 수년에 걸쳐 급격하게 변화하였으므로, 1994년에 IETF는 DNS를 근본적으로 안전하게 만들기 위한 워킹 그룹을 설립하였습니다. 이 (진행 중인) 프로젝트는 **DNSsec(DNS security)** 으로 알려져 있으며, 첫 번째 결과물은 RFC 2535에 제시되었습니다. 불행히도 DNSsec은 아직 완전히 배치되지 않아, 수많은 DNS 서버들이 여전히 스푸핑 공격에 취약합니다.

#### DNSsec의 기본 개념

DNSsec은 개념적으로 극히 단순합니다. **공개키 암호화(public-key cryptography)** 를 기반으로 합니다. 모든 DNS 존(zone)은 (그림 7-5의 의미에서) 공개/개인 키 쌍을 가집니다. DNS 서버가 보내는 모든 정보는 해당 존의 **개인키(private key)** 로 서명되므로, 수신자는 그 진위를 검증할 수 있습니다.

#### DNSsec이 제공하는 세 가지 기본 서비스

DNSsec은 세 가지 기본 서비스를 제공합니다:

1. **데이터 출처 증명 (Proof of where the data originated)**: 반환된 데이터가 해당 존의 소유자에 의해 승인되었음을 확인합니다.
2. **공개키 배포 (Public key distribution)**: 공개키를 안전하게 저장하고 검색하는 데 유용합니다.
3. **트랜잭션 및 요청 인증 (Transaction and request authentication)**: 재생 공격(playback attack)과 스푸핑 공격(spoofing attack)을 방어하는 데 필요합니다.

**비밀성(secrecy)** 은 제공되는 서비스가 아닙니다. DNS의 모든 정보는 공개적인 것으로 간주되기 때문입니다. DNSsec의 단계적 도입은 수년이 걸릴 것으로 예상되며, 보안을 인식하는 서버가 보안을 모르는 서버와 상호 운용할 수 있는 능력이 필수적이고, 이는 프로토콜을 변경할 수 없음을 의미합니다.

#### RRSet (Resource Record Sets)

DNS 레코드들은 **RRSet(Resource Record Sets)** 이라고 불리는 집합으로 그룹화됩니다. 동일한 이름, 클래스, 타입을 가진 모든 레코드가 하나의 집합으로 묶입니다. 예를 들어, DNS 이름이 기본 IP 주소와 보조 IP 주소를 가지는 경우 하나의 RRSet에 여러 A 레코드가 포함될 수 있습니다.

RRSet은 아래에서 논의할 여러 새로운 레코드 타입으로 확장됩니다. 각 RRSet은 암호학적으로 해시됩니다(예: SHA-1 사용). 이 해시는 해당 존의 **개인키**로 서명됩니다(예: RSA 사용). 클라이언트에 대한 전송 단위는 **서명된 RRSet**입니다. 서명된 RRSet을 수신하면, 클라이언트는 해당 존의 개인키로 서명되었는지 확인할 수 있습니다. 서명이 일치하면 데이터가 수락됩니다. 각 RRSet이 자체 서명을 포함하므로, RRSet은 신뢰할 수 없는 서버에서도 어디에나 캐시될 수 있으며 보안을 위협하지 않습니다.

#### DNSsec의 새로운 레코드 타입

DNSsec은 여러 새로운 레코드 타입을 도입합니다:

##### KEY 레코드

첫 번째는 **KEY 레코드**입니다. 이 레코드는 존, 사용자, 호스트 또는 기타 주체의 **공개키**, 서명에 사용된 암호화 알고리즘, 전송에 사용된 프로토콜, 그리고 기타 몇 비트를 보유합니다. 공개키는 그대로(naked) 저장됩니다. X.509 인증서는 크기가 크기 때문에 사용되지 않습니다. 알고리즘 필드는 MD5/RSA 서명의 경우 1을, 다른 조합의 경우 다른 값을 가집니다. 프로토콜 필드는 IPsec이나 다른 보안 프로토콜의 사용을 나타낼 수 있습니다.

##### SIG 레코드

두 번째 새로운 레코드 타입은 **SIG 레코드**입니다. KEY 레코드에 지정된 알고리즘에 따라 서명된 해시를 보유합니다. 서명은 RRSet의 모든 레코드에 적용되며, 존재하는 KEY 레코드는 포함하지만 SIG 레코드 자체는 제외합니다. 또한 서명의 유효 기간이 시작되는 시간, 만료 시간, 서명자의 이름 및 기타 몇 가지 항목도 보유합니다.

#### DNSsec의 키 보안 설계

DNSsec의 설계는 존의 **개인키를 오프라인으로 유지**할 수 있도록 되어 있습니다. 하루에 한두 번, 존의 데이터베이스 내용을 개인키가 위치한 연결이 끊긴 머신(예: CD-ROM)으로 수동 전송할 수 있습니다. 그곳에서 모든 RRSet에 서명할 수 있고, 이렇게 생성된 SIG 레코드를 존의 기본 서버에 다시 전달할 수 있습니다. 이 방식으로 개인키는 CD-ROM에 안전하게 저장될 수 있으며, 그날의 새로운 RRSet에 서명하기 위해 연결이 끊긴 머신에 삽입될 때를 제외하고는 잠겨 있습니다. 서명이 완료되면 키의 모든 복사본이 메모리와 디스크에서 삭제되고 CD-ROM이 안전한 곳으로 돌아갑니다. 이 절차는 전자적 보안을 물리적 보안으로 축소시키며, 이는 사람들이 다루는 방법을 이해하는 것입니다.

미리 서명된 RRSet을 사용하는 이 방법은 쿼리에 응답할 때 암호화 작업을 실시간으로 수행할 필요가 없으므로 응답 과정을 크게 가속화합니다. 단점은 DNS 데이터베이스에 모든 키와 서명을 저장하기 위해 대량의 디스크 공간이 필요하다는 것입니다. 일부 레코드는 서명으로 인해 크기가 10배 증가할 수 있습니다.

#### 공개키 검증 과정

클라이언트 프로세스가 서명된 RRSet을 받으면, 해당 존의 공개키를 적용하여 해시를 복호화하고, 해시 자체를 계산하여 두 값을 비교해야 합니다. 일치하면 데이터가 유효한 것으로 간주됩니다. 그러나 이 절차는 클라이언트가 어떻게 해당 존의 공개키를 얻는지에 대한 질문을 제기합니다. 한 가지 방법은 신뢰할 수 있는 서버에서 안전한 연결(예: IPsec 사용)을 통해 획득하는 것입니다.

그러나 실제로는, 클라이언트가 **모든 최상위 도메인의 공개키로 사전 구성(preconfigured)** 될 것으로 예상됩니다. Alice가 Bob의 웹 사이트를 방문하려면, *bob.com*의 RRSet에 대해 DNS에 요청할 수 있는데, 이 RRSet에는 Bob의 IP 주소를 포함하는 A 레코드와 Bob의 공개키를 포함하는 KEY 레코드가 있습니다. 이 RRSet은 최상위 *com* 도메인에 의해 서명되므로, Alice는 그 유효성을 쉽게 검증할 수 있습니다.

<img width="683" height="222" alt="Image" src="https://github.com/user-attachments/assets/886f16fe-cc0d-47dd-989b-10547031e364" />

이제 검증된 Bob의 공개키 사본을 가지고, Alice는 Bob이 운영하는 DNS 서버에 *www.bob.com*의 IP 주소를 요청할 수 있습니다. 이 RRSet은 Bob의 개인키로 서명되므로, Alice는 Bob이 반환한 RRSet의 서명을 검증할 수 있습니다. Trudy가 어떻게든 캐시에 거짓 RRSet을 주입하더라도, Alice는 SIG 레코드에 포함된 서명이 올바르지 않을 것이므로 그 진위 부족을 쉽게 감지할 수 있습니다.

#### 추가 보안 메커니즘

DNSsec은 또한 특정 쿼리에 대한 응답을 바인딩하는 암호화 메커니즘을 제공하여, 그림 8-47에서 Trudy가 수행한 종류의 스푸핑을 방지합니다. 이 (선택적) 위조 방지 조치는 응답자의 개인키로 서명된 쿼리 메시지의 해시를 응답에 추가합니다. Trudy는 최상위 *com* 서버의 개인키를 모르므로, Alice의 ISP가 보낸 쿼리에 대한 응답을 위조할 수 없습니다. 물론 해시된 쿼리에 대한 무효 서명 때문에 자신의 응답을 먼저 보낼 수는 있지만 거부될 것입니다.

#### 기타 레코드 타입

DNSsec은 몇 가지 다른 레코드 타입도 지원합니다. 예를 들어, **CERT 레코드**는 X.509 인증서 등을 저장하는 데 사용될 수 있습니다. 이 레코드는 일부 사람들이 DNS를 **PKI**로 전환하고 싶어하기 때문에 제공되었습니다. 이것이 실제로 일어날지는 두고 볼 일입니다. DNSsec에 대한 논의는 여기서 마치겠습니다. 자세한 내용은 RFC 2535를 참조하시기 바랍니다.

## 8.9.3 SSL -- 보안 소켓 계층 (The Secure Sockets Layer)

안전한 이름 지정은 좋은 출발이지만, 웹 보안에는 그 이상이 필요합니다. 다음 단계는 **안전한 연결(secure connections)** 입니다. 이제 안전한 연결을 어떻게 달성할 수 있는지 살펴보겠습니다. 보안과 관련된 어떤 것도 간단하지 않으며, 이것도 예외가 아닙니다.

### SSL의 탄생 배경

웹이 대중에게 처음 공개되었을 때, 처음에는 정적 페이지를 배포하는 데만 사용되었습니다. 그러나 곧 일부 회사들이 웹을 **신용카드를 통한 상품 구매**, **온라인 뱅킹**, **전자 주식 거래** 등의 금융 거래에 사용하는 아이디어를 갖게 되었습니다. 이러한 애플리케이션들은 안전한 연결에 대한 수요를 만들었습니다. 1995년, 당시 지배적인 브라우저 공급업체였던 **Netscape Communications Corp.** 은 이 수요를 충족하기 위해 **SSL(Secure Sockets Layer)** 이라는 보안 패키지를 도입하여 대응하였습니다. 이 소프트웨어와 그 프로토콜은 현재 Firefox, Safari, Internet Explorer 등에서 널리 사용되고 있어, 자세히 살펴볼 가치가 있습니다.

### SSL이 제공하는 기능

SSL은 두 소켓 사이에 안전한 연결을 구축하며, 다음을 포함합니다:

1. **클라이언트와 서버 간의 파라미터 협상 (Parameter negotiation)**
2. **클라이언트에 의한 서버 인증 (Authentication of the server by the client)**
3. **비밀 통신 (Secret communication)**
4. **데이터 무결성 보호 (Data integrity protection)**

참고로, SSL은 웹 브라우저에 국한되지 않지만, 웹 브라우저가 가장 일반적인 애플리케이션입니다. SSL은 **상호 인증(mutual authentication)** 도 제공할 수 있습니다.

### SSL의 프로토콜 스택 위치

일반적인 프로토콜 스택에서 SSL의 위치는 **그림 8-49**에 나타나 있습니다.

<img width="549" height="221" alt="Image" src="https://github.com/user-attachments/assets/2f2ac998-957a-4535-ae75-10a2b4d38a33" />

사실상 SSL은 **응용 계층과 전송 계층 사이에 삽입된 새로운 계층**으로, 브라우저의 요청을 받아서 서버로의 전송을 위해 TCP로 내려보냅니다. 안전한 연결이 확립되면, SSL의 주요 작업은 **압축(compression)** 과 **암호화(encryption)** 를 처리하는 것입니다. HTTP가 SSL 위에서 사용되면, 표준 HTTP 프로토콜임에도 불구하고 **HTTPS(Secure HTTP)** 라고 합니다. 때로는 포트 80 대신 새로운 포트 **(443)** 에서 사용 가능합니다.

### SSL 버전과 옵션

SSL 프로토콜은 여러 버전을 거쳤습니다. 아래에서는 가장 널리 사용되는 **버전 3**만 논의합니다. SSL은 다양한 옵션을 지원합니다. 이러한 옵션에는 **압축의 유무**, 사용할 **암호화 알고리즘**, 그리고 **암호화 수출 제한**과 관련된 일부 사항이 포함됩니다.

수출 제한과 관련하여, 마지막 옵션은 주로 연결의 양 끝이 미국 내에 있을 때만 심각한 암호화가 사용되도록 하기 위한 것입니다. 다른 경우에는 키가 **40비트**로 제한되는데, 암호학자들은 이를 일종의 농담으로 여깁니다. Netscape는 미국 정부로부터 수출 허가를 받기 위해 이 제한을 포함시킬 수밖에 없었습니다.

### SSL의 두 가지 하위 프로토콜

SSL은 두 개의 하위 프로토콜로 구성됩니다: **안전한 연결을 확립하기 위한 것**과 **이를 사용하기 위한 것**입니다.

#### 연결 확립 하위 프로토콜 (Connection Establishment Subprotocol)

안전한 연결이 어떻게 확립되는지 먼저 살펴보겠습니다. **연결 확립 하위 프로토콜**은 **그림 8-50**에 나타나 있습니다.

<img width="710" height="485" alt="Image" src="https://github.com/user-attachments/assets/31594357-0571-4ed8-afaa-47939549b944" />

각 메시지의 상세 설명은 다음과 같습니다:

1. **메시지 1**: Alice가 Bob에게 연결 확립 요청을 보내며 시작합니다. 요청에는 Alice가 가진 SSL 버전과 압축 및 암호화 알고리즘에 대한 선호 설정이 지정됩니다. 나중에 사용할 **논스(nonce) R_A**도 포함됩니다.

2. **메시지 2**: Bob의 차례입니다. Bob은 Alice가 지원할 수 있는 다양한 알고리즘 중에서 선택하고 자신의 논스 **R_B**를 보냅니다.

3. **메시지 3**: Bob은 자신의 공개키를 포함하는 **인증서(certificate)** 를 보냅니다. 이 인증서가 잘 알려진 인증기관에 의해 서명되지 않은 경우, 하나까지 거슬러 올라갈 수 있는 **인증서 체인(chain of certificates)** 도 보냅니다. Alice의 것을 포함한 모든 브라우저에는 약 100개의 공개키가 사전 탑재(preloaded)되어 있으므로, Bob이 이들 중 하나에 앵커된 체인을 확립할 수 있다면, Alice는 Bob의 공개키를 검증할 수 있을 것입니다. 이 시점에서 Bob은 다른 메시지(예: Alice의 공개키 인증서 요청)를 보낼 수도 있습니다.

4. **메시지 4**: Bob이 완료되면 Alice에게 차례임을 알리기 위해 이 메시지를 보냅니다.

5. **메시지 5**: Alice는 384비트의 랜덤 **프리마스터 키(premaster key)** 를 선택하고 Bob의 공개키로 암호화하여 Bob에게 보내어 응답합니다. 데이터를 암호화하는 데 사용되는 실제 세션 키는 프리마스터 키와 양쪽 논스를 복잡한 방식으로 결합하여 도출됩니다. 메시지 5가 수신된 후, Alice와 Bob 모두 세션 키를 계산할 수 있습니다.

6. **메시지 6**: Alice는 Bob에게 새로운 암호로 전환하도록 지시합니다.

7. **메시지 7**: Alice는 확립 하위 프로토콜이 완료되었음을 알립니다.

8. **메시지 8**: Bob은 Alice를 확인하고 응답합니다.

9. **메시지 9**: Bob은 확립 하위 프로토콜이 완료되었음을 알립니다.

#### 인증의 비대칭성

Alice가 Bob이 누구인지 알지만, Bob은 Alice가 누구인지 모릅니다(Alice가 공개키와 해당 인증서를 가지고 있지 않는 한, 이는 개인 사용자에게는 드문 상황). 따라서 Bob의 첫 번째 메시지는 이전에 확립된 로그인 이름과 패스워드를 사용하여 로그인하라는 요청일 수 있습니다. 로그인 프로토콜은 SSL의 범위 밖입니다. 로그인이 완료되면, 어떤 방법으로든, 데이터 전송이 시작될 수 있습니다.

### SSL이 지원하는 암호화 알고리즘

앞서 언급한 바와 같이, SSL은 다수의 암호화 알고리즘을 지원합니다:

| 암호화 조합 | 용도 | 설명 |
|------------|------|------|
| **Triple DES** (3개의 별도 키) + **SHA-1** | 뱅킹 등 최고 보안 | 가장 강력한 조합이지만 비교적 느림 |
| **RC4** (128비트 키) + **MD5** | 일반 전자상거래 | RC4는 128비트 키를 시드로 사용하여 더 큰 내부 수로 확장한 다음 키스트림을 생성. 키스트림을 평문과 XOR하여 고전적인 스트림 암호 제공 |
| **RC4** (128비트 키, 88비트 공개) | 수출 버전 | 128비트 키 중 88비트가 공개되어 암호 해독이 쉬움 |

#### Triple DES + SHA-1
가장 강력한 조합은 암호화에 세 개의 별도 키를 사용하는 **Triple DES**와 메시지 무결성을 위한 **SHA-1**을 사용합니다. 이 조합은 비교적 느리므로, 최고의 보안이 필요한 뱅킹 및 기타 애플리케이션에 주로 사용됩니다.

#### RC4 + MD5
일반적인 전자상거래 애플리케이션에서는 암호화에 **128비트 키를 사용하는 RC4**가, 메시지 인증에는 **MD5**가 사용됩니다. RC4는 128비트 키를 시드로 사용하여 이를 훨씬 더 큰 내부 수로 확장한 다음, 이 내부 수를 사용하여 **키스트림(keystream)** 을 생성합니다. 키스트림을 평문과 **XOR**하여 그림 8-14에서 본 것과 같은 고전적인 스트림 암호를 제공합니다. 수출 버전도 128비트 키를 사용하는 RC4를 사용하지만, 88비트가 공개되어 암호 해독이 쉬워집니다.

### 데이터 전송 하위 프로토콜 (Data Transport Subprotocol)

실제 데이터 전송을 위해, **그림 8-51**에 표시된 두 번째 하위 프로토콜이 사용됩니다.

<img width="634" height="423" alt="Image" src="https://github.com/user-attachments/assets/cc2730ab-76c2-4873-a8e4-bb6eaaf05ee4" />

> 1. **단편화 (Fragmentation)**: 브라우저의 메시지가 최대 16KB의 단위(unit)로 분할됩니다.
> 2. **압축 (Compression)**: 압축이 활성화된 경우, 각 단위가 별도로 압축됩니다.
> 3. **MAC 추가 (MAC added)**: 두 논스와 프리마스터 키에서 도출된 비밀키가 압축된 텍스트와 연결되고, 합의된 해싱 알고리즘(보통 MD5)으로 해시한 결과가 **MAC(메시지 인증 코드)** 가 됩니다. 이 해시가 각 조각에 추가됩니다.
> 4. **암호화 (Encryption)**: 압축된 조각과 MAC이 합의된 대칭 암호화 알고리즘(보통 RC4 키스트림과 XOR)으로 암호화됩니다.
> 5. **헤더 추가 (Header added)**: 마지막으로, 조각 헤더가 부착되고 조각이 TCP 연결을 통해 전송됩니다.

### SSL의 보안 주의사항

RC4에 쉽게 암호 분석될 수 있는 **약한 키(weak keys)** 가 있다는 것이 입증되었으므로(Fluhrer et al., 2001), RC4를 사용하는 SSL의 보안은 불안정한 기반 위에 있습니다. 사용자가 암호 스위트를 선택할 수 있는 브라우저는 항상 **168비트 키를 사용하는 Triple DES와 SHA-1**을 사용하도록 설정해야 합니다. RC4와 MD5보다 느리더라도 말입니다. 더 나은 것은, SSL의 후속인 TLS를 지원하는 브라우저로 업그레이드하는 것입니다.

SSL의 문제점 중 하나는 주체(principals)가 인증서를 가지고 있지 않을 수 있고, 가지고 있더라도 사용 중인 키와 항상 일치하는지 검증하지 않는다는 점입니다.

### TLS (Transport Layer Security)

1996년에 Netscape Communications Corp.은 표준화를 위해 SSL을 IETF에 넘겼습니다. 그 결과가 **TLS(Transport Layer Security)** 입니다. RFC 5246에 기술되어 있습니다.

TLS는 **SSL 버전 3** 위에 구축되었습니다. SSL에 대한 변경 사항은 비교적 작았지만, SSL 버전 3과 TLS가 상호 운용할 수 없을 정도로는 충분히 달랐습니다. 예를 들어, 프리마스터 키와 논스에서 세션 키를 도출하는 방식이 변경되어 키를 더 강하게(즉, 암호 분석하기 더 어렵게) 만들었습니다.

이러한 비호환성 때문에, 대부분의 브라우저는 양쪽 프로토콜을 모두 구현하며, 필요시 TLS가 SSL로 폴백(falling back)합니다. 이를 **SSL/TLS**라고 합니다. 첫 번째 TLS 구현은 1999년에 나타났으며 2008년 8월에 버전 1.2가 정의되었습니다. TLS는 더 강력한 암호 스위트(특히 **AES**)에 대한 지원을 포함합니다. SSL은 시장에서 강한 위치를 유지하고 있지만, TLS가 점진적으로 이를 대체할 것으로 예상됩니다.

## 8.9.4 모바일 코드 보안 (Mobile Code Security)

이름 지정과 연결은 웹 보안과 관련된 두 가지 관심 영역입니다. 하지만 그 외에도 더 있습니다. 초기에 웹 페이지가 정적 HTML 파일이었을 때는 실행 가능한 코드를 포함하지 않았습니다. 이제 웹 페이지는 **Java 애플릿(Java applets)**, **ActiveX 컨트롤(ActiveX controls)**, **JavaScript** 등의 작은 프로그램을 종종 포함합니다. 이러한 **모바일 코드(mobile code)** 를 다운로드하고 실행하는 것은 명백히 대규모 보안 위험이므로, 이를 최소화하기 위해 다양한 방법이 고안되었습니다.

이제 모바일 코드에 의해 제기되는 몇 가지 이슈와 이에 대처하기 위한 접근 방식을 간략히 살펴보겠습니다.

---

### Java 애플릿 보안 (Java Applet Security)

**Java 애플릿(Java applets)** 은 **JVM(Java Virtual Machine)** 이라는 스택 지향 머신 언어로 컴파일된 작은 Java 프로그램입니다. 웹 페이지에 배치되어 페이지와 함께 다운로드될 수 있습니다. 페이지가 로딩된 후, 애플릿은 **그림 8-52**에 설명된 것처럼 브라우저 내부의 JVM 인터프리터에 삽입됩니다.

<img width="508" height="295" alt="Image" src="https://github.com/user-attachments/assets/2a5d3ba8-b1a6-4058-bbc8-d63af8ac225f" />

> - 최하위 주소에 웹 브라우저와 인터프리터가 위치합니다.
> - **신뢰할 수 있는 애플릿(trusted applet)** 은 브라우저 근처에 위치합니다.
> - **신뢰할 수 없는 애플릿(untrusted applet)** 은 **샌드박스(sandbox)** 내부에 격리되어 위치합니다.
> - 주소 공간의 상위에는 0xFFFFFFFF까지 확장됩니다.

#### 인터프리티드 코드의 장점

컴파일된 코드 대신 인터프리티드(interpreted) 코드를 실행하는 장점은 **모든 명령어가 실행 전에 인터프리터에 의해 검사**된다는 것입니다. 이는 인터프리터에게 **명령어의 주소가 유효한지 확인**할 기회를 제공합니다. 또한 **시스템 콜(system calls)** 도 포착되어 해석됩니다. 이러한 호출을 어떻게 처리할지는 **보안 정책(security policy)** 의 문제입니다.

#### 신뢰/비신뢰 애플릿

예를 들어, 애플릿이 신뢰할 수 있는 경우(예: 로컬 디스크에서 온 경우), 해당 시스템 콜은 의심 없이 수행될 수 있습니다. 그러나 애플릿이 신뢰할 수 없는 경우(예: 인터넷을 통해 들어온 경우), **샌드박스(sandbox)** 라고 불리는 것 안에 캡슐화되어 시스템 리소스를 사용하려는 시도가 차단될 수 있습니다.

#### 보안 모니터

애플릿이 시스템 리소스를 사용하려고 할 때, 호출은 **보안 모니터(security monitor)** 에 전달되어 승인을 받습니다. 모니터는 로컬 보안 정책에 비추어 호출을 검사하고 허용 또는 거부 결정을 내립니다. 이 방식으로 애플릿이 일부 리소스에는 접근하되 전부에는 접근하지 못하도록 할 수 있습니다. 불행히도, 현실은 이 보안 모델이 잘 작동하지 않고 버그가 항상 나타난다는 것입니다.

---

### ActiveX

**ActiveX 컨트롤(ActiveX controls)** 은 웹 페이지에 내장될 수 있는 **x86 바이너리 프로그램**입니다. 이들 중 하나가 발견되면, 실행 여부를 확인하는 검사가 수행되고, 테스트를 통과하면 실행됩니다. ActiveX 컨트롤은 **해석되거나 샌드박스화되지 않으며**, 다른 사용자 프로그램과 동일한 권한을 가지므로 잠재적으로 큰 피해를 줄 수 있습니다. 따라서 모든 보안은 ActiveX 컨트롤을 **실행할지 여부의 결정**에 달려 있습니다. 돌이켜 보면, 전체 아이디어는 거대한 보안 허점입니다.

#### 코드 서명 (Code Signing)

Microsoft가 이 결정을 내리기 위해 선택한 방법은 **코드 서명(code signing)** 이라는 개념에 기반합니다. 각 ActiveX 컨트롤에는 **디지털 서명(digital signature)** 이 동반됩니다 -- 이것은 공개키 암호화를 사용하여 작성자가 서명한 코드의 해시입니다. ActiveX 컨트롤이 나타나면, 브라우저는 먼저 서명을 검증하여 전송 중에 변조되지 않았는지 확인합니다. 서명이 올바르면, 브라우저는 내부 테이블을 확인하여 프로그램의 작성자가 신뢰할 수 있는지, 또는 신뢰할 수 있는 작성자까지의 신뢰 체인이 있는지 확인합니다. 작성자가 신뢰할 수 있으면 프로그램이 실행되고, 그렇지 않으면 실행되지 않습니다. ActiveX 컨트롤을 검증하기 위한 Microsoft의 시스템을 **Authenticode**라고 합니다.

#### Java 접근법과 ActiveX 접근법의 비교

Java와 ActiveX 접근법을 대조하는 것이 유용합니다:

| 특성 | Java 접근법 | ActiveX 접근법 |
|------|-----------|--------------|
| **작성자 확인** | 시도하지 않음 | 코드 서명으로 확인 |
| **런타임 모니터링** | 인터프리터가 허용되지 않는 행위를 차단 | 런타임 모니터링 없음 |
| **실행 방식** | 인터프리터에서 실행 | 네이티브 코드로 직접 실행 |
| **보안 모델** | 코드가 무엇을 *하는지* 확인 | 코드가 *어디서 왔는지* 확인 |
| **악성 코드 대응** | 샌드박스로 제한 | 신뢰된 경우 제한 없이 실행 |

Java 접근법에서는 애플릿을 누가 작성했는지 확인하려는 시도가 없습니다. 대신, 런타임 인터프리터가 머신 소유자가 애플릿이 하면 안 된다고 말한 것을 하지 않도록 합니다. 반면, 코드 서명에서는 모바일 코드의 **런타임 행동을 모니터링하려는 시도가 없습니다**. 신뢰할 수 있는 소스에서 왔고 전송 중에 수정되지 않았다면, 그냥 실행됩니다. 코드가 악의적인지 여부를 확인하려는 시도는 없습니다. 원래 프로그래머가 하드 디스크를 포맷하고 플래시 ROM을 지우도록 코드를 *의도*했다면, 프로그래머가 신뢰할 수 있는 것으로 인증되었다면 그 코드가 실행되어 컴퓨터를 파괴합니다(브라우저에서 ActiveX 컨트롤이 비활성화되어 있지 않는 한).

#### 코드 서명의 위험성

많은 사람들은 알 수 없는 소프트웨어 회사를 신뢰하는 것이 무서운 일이라고 생각합니다. 이 문제를 보여주기 위해, 시애틀의 한 프로그래머가 소프트웨어 회사를 설립하고 신뢰할 수 있다는 인증을 받았습니다(이는 쉬운 일). 그런 다음 머신을 깔끔하게 종료(clean shutdown)하는 ActiveX 컨트롤을 작성하고 이를 널리 배포하였습니다. 많은 머신이 종료되었지만, 재부팅하면 되므로 실질적인 피해는 없었습니다. 그는 단지 문제를 세상에 알리려고 했을 뿐입니다. 공식적인 대응은 이 특정 ActiveX 컨트롤에 대한 인증서를 폐기하는 것이었고, 이로 인해 잠깐의 창피함은 끝났지만, 근본적인 문제는 악의적인 프로그래머가 악용할 수 있도록 여전히 남아 있습니다(Garfinkel with Spafford, 2002). 모바일 코드를 작성할 수 있는 수천 개의 소프트웨어 회사를 감시할 방법이 없으므로, 코드 서명 기법은 재앙이 일어나기를 기다리고 있는 것입니다.

### JavaScript

**JavaScript**는 공식적인 보안 모델이 없지만, 보안이 취약한 구현의 긴 역사를 가지고 있습니다. 각 벤더가 서로 다른 방식으로 보안을 처리합니다. 예를 들어, Netscape Navigator 버전 2는 Java 모델과 유사한 것을 사용했지만, 버전 4에서는 코드 서명 모델로 전환하였습니다.

근본적인 문제는 **외부 코드가 자신의 머신에서 실행되도록 허용하는 것은 문제를 자초하는 것**이라는 점입니다. 보안의 관점에서 보면, 이는 도둑을 집에 초대하고 부엌에서 나갈 수 없도록 감시하는 것과 같습니다. 예상치 못한 일이 발생하고 잠시 주의가 분산되면, 나쁜 일이 일어날 수 있습니다. 여기서의 긴장은 모바일 코드가 화려한 그래픽과 빠른 상호작용을 가능하게 하고, 많은 웹 사이트 디자이너들은 보안보다 이것이 훨씬 더 중요하다고 생각한다는 것입니다. 특히 다른 사람의 머신이 위험에 처할 때 말입니다.

### 브라우저 확장 기능 (Browser Extensions)

웹 페이지에 코드를 포함하는 것뿐만 아니라, **브라우저 확장 기능(browser extensions)**, **애드온(add-ons)**, **플러그인(plug-ins)** 이라는 활발한 시장이 있습니다. 이들은 웹 브라우저의 기능을 확장하는 컴퓨터 프로그램입니다.

- **플러그인(Plug-ins)**: PDF나 Flash 애니메이션 같은 특정 유형의 콘텐츠를 해석하거나 표시하는 기능을 제공합니다.
- **확장 기능(Extensions)과 애드온(Add-ons)**: 더 나은 비밀번호 관리, 페이지 북마크, 또는 관련 상품의 쉬운 쇼핑 등 새로운 브라우저 기능을 제공합니다.

#### 설치의 용이성과 위험

확장 기능, 애드온 또는 플러그인을 설치하는 것은 브라우징 중 원하는 것을 발견하고 링크를 따라가 프로그램을 설치하는 것만큼 간단합니다. 이 행위로 인해 인터넷을 통해 코드가 다운로드되어 브라우저에 설치됩니다. 이러한 프로그램들은 모두 브라우저에 따라 다른 프레임워크에 맞추어 작성되었습니다. 그러나 대략적으로 말하면, 이들은 향상되고 있는 **브라우저의 신뢰된 컴퓨팅 기반(trusted computing base)** 의 일부가 됩니다. 즉, 설치된 코드에 버그가 있으면 **전체 브라우저가 침해**될 수 있습니다.

#### 두 가지 명백한 실패 모드

1. **악의적 프로그램**: 프로그램이 악의적으로 행동할 수 있습니다. 예를 들어, 개인 정보를 수집하여 원격 서버로 보낼 수 있습니다. 브라우저가 아는 한, 사용자가 바로 그 목적을 위해 확장 기능을 설치한 것입니다.

2. **새로운 콘텐츠 유형의 취약성**: 플러그인은 브라우저에 새로운 유형의 콘텐츠를 해석하는 기능을 부여합니다. 이 콘텐츠는 종종 완전한 프로그래밍 언어 자체입니다. **PDF**와 **Flash**가 좋은 예입니다. 사용자가 브라우저에서 PDF와 Flash 콘텐츠를 볼 때, 플러그인이 PDF와 Flash 코드를 실행합니다. 그 코드는 안전해야 하지만, 종종 악용할 수 있는 취약점이 있습니다.

이 모든 이유로, 애드온과 플러그인은 **필요할 때만, 그리고 신뢰할 수 있는 벤더에서만** 설치해야 합니다.

### 바이러스 (Viruses)

**바이러스(Viruses)** 는 또 다른 형태의 모바일 코드입니다. 위의 예시들과 달리, 바이러스는 전혀 초대받지 않습니다. 바이러스와 일반 모바일 코드의 차이점은 바이러스는 **스스로를 재생산하도록 작성**되었다는 것입니다.

바이러스는 웹 페이지, 이메일 첨부 파일, 또는 다른 방법을 통해 도착하며, 보통 디스크의 실행 가능한 프로그램을 감염시키는 것으로 시작합니다. 이러한 프로그램 중 하나가 실행되면 제어권이 바이러스로 이전되고, 바이러스는 보통 자신을 다른 머신으로 전파하려고 시도합니다. 예를 들어 피해자의 이메일 주소록에 있는 모든 사람에게 자신의 복사본을 이메일로 보내는 방식입니다. 일부 바이러스는 하드 디스크의 부트 섹터를 감염시켜, 머신이 부팅될 때 바이러스가 실행되도록 합니다.

바이러스는 인터넷에서 거대한 문제가 되었으며 수십억 달러 상당의 피해를 입혔습니다. 명백한 해결책은 없습니다. 아마도 안전한 마이크로커널과 사용자, 프로세스, 리소스의 철저한 구획화(compartmentalization)를 기반으로 한 완전히 새로운 세대의 운영 체제가 도움이 될 수 있을 것입니다.
